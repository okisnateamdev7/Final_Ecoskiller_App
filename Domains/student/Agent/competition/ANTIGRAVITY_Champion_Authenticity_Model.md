## SECTION ACAM-5 — AUTHENTICITY ALGORITHM SPECIFICATIONS

### ACAM-5.1 — Composite Authenticity Score Algorithm

```
INPUTS:
  identity_score           (from IdentityVerificationEngine)
  behavior_score           (from BehavioralAnomalyModel)
  match_score              (from MatchActivityVerifier)
  score_integrity_score    (from CollusionDetectionEngine)
  replay_score             (from ReplayIntegrityVerifier)
  consistency_score        (from CareerConsistencyEngine)

WEIGHTS (v1.0 — immutable without version bump):
  w_identity    = 0.25
  w_behavior    = 0.25
  w_match       = 0.15
  w_score       = 0.15
  w_replay      = 0.10
  w_consistency = 0.10

FORMULA:
  authenticity_score = (
    w_identity    × identity_score    +
    w_behavior    × behavior_score    +
    w_match       × match_score       +
    w_score       × score_integrity_score +
    w_replay      × replay_score      +
    w_consistency × consistency_score
  )

THRESHOLD ACTIONS:
  authenticity_score ≥ 0.90  → CLEAN — eligible for Championship Trust Seal
  0.75 ≤ score < 0.90        → MONITORING — flagged for passive review
  0.55 ≤ score < 0.75        → FLAGGED — active case review required
  score < 0.55               → UNDER_REVIEW — championship eligibility suspended
                                pending Governance Board ruling

GOVERNANCE RULE:
  AI produces the score. AI raises the flag.
  Governance Board decides eligibility. Never AI alone.
  Belt promotion blocked ONLY by explicit Governance Board ruling, not by AI score alone.
```

### ACAM-5.2 — Behavioral Anomaly Detection Model (LSTM)

```
MODEL TYPE: Long Short-Term Memory (LSTM) Sequence Classifier

INPUT SEQUENCE:
  Per-match behavioral event stream from BehavioralEventLog
  Window: full match session (variable length, padded)
  Features per event: event_type_encoded, timestamp_delta_ms,
                      response_length, cadence_variance,
                      session_fingerprint_consistency

TRAINING DATA:
  Labeled historical matches: {AUTHENTIC, SUSPICIOUS, CONFIRMED_FRAUD}
  Minimum 10,000 labeled sessions before production deployment
  Retrained quarterly with new confirmed-fraud cases added

OUTPUT:
  anomaly_probability: 0.0 – 1.0
  triggered_signal_codes: [ BEHAVIOR_EVENT_TAXONOMY codes ]
  confidence: 0.0 – 1.0

DECISION RULE:
  anomaly_probability ≥ 0.80 AND confidence ≥ 0.70
    → behavior_score = 1 - anomaly_probability (low behavior_score = bad)
    → BehavioralEventLog.anomaly_flag = TRUE
    → Route to FlagRouter

MINIMUM DATA RULE:
  < 5 match sessions → model returns behavior_score = NULL (insufficient data)
  NULL sub-scores excluded from composite with weight redistributed proportionally
  No authenticity penalties applied on insufficient data alone
```

### ACAM-5.3 — Collusion Detection Engine (Graph Model)

```
MODEL TYPE: Weighted Directed Graph + Statistical Outlier Detection

GRAPH CONSTRUCTION:
  Nodes: User accounts
  Edges: CollusionGraphEdge records
  Edge weight: reciprocal_score_count / total_shared_matches

DETECTION ALGORITHM:
  Step 1 — Build score exchange matrix for all peer-scoring pairs
  Step 2 — Calculate reciprocity ratio:
    reciprocity_ratio = (A→B avg score / global avg score) ×
                        (B→A avg score / global avg score)
  Step 3 — Calculate suspicion score:
    suspicion_score = normalize(reciprocity_ratio × frequency_weight)
  Step 4 — Identify cliques:
    IF 3+ users form a mutually high-scoring clique → clique_flag = TRUE

THRESHOLDS:
  suspicion_score ≥ 0.75 → CollusionGraphEdge.status = FLAGGED
  clique_flag = TRUE → GovernanceCaseDossier.case_type = COLLUSION

CONSTRAINTS:
  Algorithm runs post-match and post-tournament (not real-time)
  Minimum 10 shared scoring events before edge is scored
  Mentor-assigned scores excluded from collusion graph (separate audit path)
```

### ACAM-5.4 — Identity Verification Engine

```
VERIFICATION LAYERS (ALL REQUIRED):

  Layer 1 — Account Verification (from Ecoskiller Trust System):
    institution_email verified → verified_status = TRUE
    Mandatory before any ranked match participation

  Layer 2 — Device Registry Validation:
    Device registered at first login
    Unregistered device during active tournament
      → IDENTITY_LOGIN_DEVICE_CHANGE event raised
      → Match paused for re-authentication

  Layer 3 — Session Fingerprint Consistency:
    Fingerprint established at session start (typing rhythm, cadence)
    Mid-session fingerprint drift >2σ from baseline
      → IDENTITY_RESPONSE_CADENCE_SHIFT event raised
      → identity_score decrement applied

  Layer 4 — Multi-Account Detection:
    Device fingerprint + IP zone hash cross-checked against all active accounts
    Shared fingerprint across 2+ accounts
      → IDENTITY_MULTI_ACCOUNT_DETECTED raised
      → Both accounts flagged, matches suspended pending governance review

  Layer 5 — Simultaneous Session Detection:
    Per user: only 1 active match session permitted at any time
    Duplicate session detected → earlier session terminated + event raised

SCORING:
  All layers clean → identity_score = 1.0
  Each failed layer applies weighted penalty:
    Layer 1 unverified     → identity_score × 0.50 (significant penalty)
    Layer 2 device anomaly → identity_score × 0.85
    Layer 3 cadence shift  → identity_score × 0.90
    Layer 4 multi-account  → identity_score = 0.00 (hard zero — auto flag)
    Layer 5 dual session   → identity_score = 0.00 (hard zero — auto flag)
```

### ACAM-5.5 — Replay Integrity Verifier

```
PROCESS:
  Step 1 — On match end, ReplayEngine emits replay_processed event
  Step 2 — ACAM taps replay stream, calculates SHA-256 hash per 30-second segment
  Step 3 — Hash chain root = Merkle root of all segment hashes
  Step 4 — Store in ReplayIntegrityCertificate

  Step 5 — Verification (on demand or governance review):
    Recalculate segment hashes from stored replay
    Compare to stored hash chain
    Any mismatch → REPLAY_HASH_MISMATCH raised
    Gap in timeline → REPLAY_SEGMENT_MISSING raised

  Step 6 — Duration validation:
    |replay_duration - match_declared_duration| > 5% → REPLAY_TIMING_INCONSISTENCY

CHAIN CUSTODY RULE:
  Hash chain root stored immutably in PostgreSQL at match completion
  No post-hoc modification of replay files without new hash + governance approval
  Original replay files stored in MinIO with server-side object lock (WORM policy)
```

### ACAM-5.6 — Career Consistency Engine

```
INPUT:
  Full match history (all scores, all belts, all tournaments)
  Skill growth velocity (from AntiGravity ACFE)
  Drill completion history
  Mentor feedback history

ALGORITHM:
  Step 1 — Fit expected_performance_curve using historical data
    (Linear regression on rolling 30-match window, belt-adjusted)
  Step 2 — Calculate residual for each performance event
  Step 3 — Flag if:
    Single event residual > 3σ → CONSISTENCY_SUDDEN_RATING_SPIKE
    Skill vector changes >40% within 14 days without drill evidence
      → CONSISTENCY_SKILL_VECTOR_DISCONTINUITY
    New belt level performance ≥ 25% above trajectory top bound
      → CONSISTENCY_BELT_JUMP_ANOMALY

PARENT INTELLIGENCE INTEGRATION:
  Career consistency data fed into ParentPredictiveAI as authenticity_signal
  Genuine breakthroughs vs. suspicious spikes distinguished by:
    - Drill correlation (real breakthrough has drill evidence)
    - Mentor annotation (mentor confirms improvement)
    - Temporal plausibility (improvement velocity realistic for age/domain)
  Parent shown: "Notable performance gain detected — under review for authenticity"
  Parent NOT shown: "Your child is suspected of cheating"
  Language is factual, non-accusatory, informational
```

```
      - Relevant BehavioralEventLog entries

      - ReplayIntegrityCertificate(s)
      - Flagged event codes with timestamps
    AI generates: summary narrative for governance reviewer
    AI does NOT generate: verdict, guilt probability score shown to anyone

  STEP 3 — ASSIGNMENT
    Case assigned to available Governance Board member
    Conflict check: Board member may not review case if they have scored
    the subject user in the last 90 days

  STEP 4 — REVIEW (HUMAN)
    Board member reviews evidence bundle
    May request: additional replay segments, mentor statements, peer statements
    May NOT request: AI-generated verdict statement

  STEP 5 — RESOLUTION (HUMAN-ONLY)
    Board member records decision:
      RESOLVED_CLEAN       → No action. All results stand.
      RESOLVED_PENALIZED   → Specific matches voided. Rating adjusted.
                             Belt blocked or revoked. Account flagged.
      RESOLVED_CLEARED     → Insufficient evidence. No action. Monitoring continues.

  STEP 6 — NOTIFICATION
    Subject user notified: plain language, rights-preserving
    Parent notified: via ParentAuthenticityAlert (if subject is minor or parent-linked)
    Mentor notified: outcome only, no evidence detail

  STEP 7 — SEAL DECISION
    If RESOLVED_CLEAN + all preconditions met → Board issues Trust Seal
    If RESOLVED_PENALIZED → Seal blocked or revoked

  STEP 8 — APPEAL
    Subject has 14 days to file appeal
    Appeal reviewed by separate board panel (not original reviewer)
    Appeal decision is final
    All decisions immutably logged

PROHIBITED ACTIONS AT ANY STEP:
  AI may not issue verdicts
  AI may not send accusatory messages to users
  AI may not post to public-facing profiles during open case
  No partial evidence disclosure to subject during active review
  No cross-case contamination (separate dossiers for separate incidents)
```
