├─ Definition: People currently training (not yet certified)
├─ Measurement:
│  ├─ Enrolled in skill track: In-progress

│
├─ Supply forecast calculation:
│  └─ Future_Supply = Current_Pipeline × Completion_Rate
│
└─ Example:
   ├─ Python: 600 in training
   ├─ Expected completion: 60% (360 new Yellow Belt/year)
   └─ Supply added: ~30/month to active pool

COMPONENT 3: SKILL ATTRITION (Supply loss)
├─ Definition: People leaving skill domain (retirement, career change)
├─ Measurement:
│  ├─ Match activity drop-off (no matches for 6+ months)
│  ├─ Career path changes (tracking job transitions)
│  ├─ Retirement (identified via profile data)
│  └─ Skill obsolescence (switching to new tech)
│
├─ Rates:
│  ├─ Novice: 15%/year attrition (high, unproven skills)
│  ├─ Intermediate: 8%/year (committed practitioners)
│  ├─ Expert: 3%/year (low, valuable skills)
│  └─ Master: 1%/year (very rare, dedicated)
│
└─ Example:
   ├─ Python Novices: 500 × 15% = 75/year lost
   ├─ Python Expert: 300 × 3% = 9/year lost
   └─ Net attrition: ~84/year, dampened by inflow

COMPONENT 4: SKILL ADVANCEMENT (Internal supply rebalancing)
├─ Definition: People advancing within skill (Novice → Expert)
├─ Measurement:
│  ├─ Belt progression rates (track by skill)
│  ├─ Time-in-skill (experience accumulation)
│  └─ Performance trajectory (improvement over time)
│
├─ Rates (% advancing per period):
│  ├─ White → Yellow: 60% of engaged novices (8–12 weeks)
│  ├─ Yellow → Green: 50% of intermediate (12–20 weeks)
│  ├─ Green → Blue: 40% of proficient (20–32 weeks)
│  ├─ Blue → Black: 20% of experts (32–52 weeks)
│  └─ Total pipeline: ~10% per month advance to next level
│
└─ Impact: Shifts supply distribution (fewer novices, more experts over time)

TOTAL SUPPLY FORECAST:
├─ Current_Supply = Active_Practitioners_Count
├─ Monthly_Inflow = Training_Completions + Returning_Practitioners
├─ Monthly_Outflow = Attrition + Career_Switches
├─ Monthly_Advancement = (Higher_Level_Gains - Lower_Level_Losses)
├─ Forecast_Supply = Current + (Inflow - Outflow + Advancement)
└─ Horizon: 3-month, 6-month, 12-month, 24-month forecasts

SUPPLY ELASTICITY:
├─ Definition: How much does training supply respond to demand increases?
├─ Model: Each 10-point demand increase → +5% enrollment (elastic)
├─ Lag time: 2–3 months (people don't immediately enroll)
├─ Saturation point: Limited number of active learners
├─ Implication: Demand shocks take months to resolve via supply response
```

## 🔒 SECTION 4 — DEMAND FORECASTING MODELS

### 4.1 Time-Series Forecasting (LOCKED MODELS)

```
FORECASTING METHODOLOGY:
═════════════════════════════════════════════════════════════════

MODEL 1: EXPONENTIAL SMOOTHING (Simple, established)
├─ Best for: Stable trends with gradual changes
├─ Formula: Forecast = α × Recent_Demand + (1−α) × Prior_Forecast
│  ├─ α (smoothing factor): 0.3 (balance recent data with history)
│  └─ Forecast confidence: 85% (for 3-month horizon)
│
├─ Application:
│  ├─ SQL Demand: Stable at 70/100, minor growth
│  ├─ 3-month forecast: 71/100 (slight increase)
│  ├─ Trend: Mature, stable skill
│  └─ Recommendation: Maintain capacity
│
└─ Limitations:
   ├─ Cannot capture sudden shifts
   ├─ Assumes trend continuation
   └─ Poor for emerging/declining skills

MODEL 2: ARIMA (AutoRegressive Integrated Moving Average)
├─ Best for: Seasonal patterns + trends
├─ Components:
│  ├─ AR: Autoregressive (past values influence future)
│  ├─ I: Integrated (differencing to remove trends)
│  ├─ MA: Moving Average (smoothing random fluctuations)
│  └─ ARIMA(p,d,q) parameters tuned per skill
│
├─ Application:
│  ├─ Python Demand: Trending upward with seasonality
│  ├─ 12-month forecast: Account for summer dips, Jan peaks
│  ├─ Trend: Growing technology skill
│  └─ Recommendation: Gradual capacity increase (8%/quarter)
│
└─ Advantages:
   ├─ Captures seasonal patterns
   ├─ Handles trends better than exponential smoothing
   └─ Good for 3–12 month horizons

MODEL 3: REGRESSION (Causal factors)
├─ Best for: Understanding drivers of demand
├─ Predictor variables (causal factors):
│  ├─ Job market health (unemployment rate)
│  ├─ Industry growth (sector-specific GDP)
│  ├─ Technology cycles (Gartner hype cycle phase)
│  ├─ Venture funding (startup creation)
│  └─ Salary trends (compensation data)
│
├─ Formula: Demand = β₀ + β₁×(Unemployment) + β₂×(Industry_Growth) + ...
│  ├─ β coefficients: Estimated from historical data
│  ├─ R-squared: Goodness of fit (target: >0.70)
│  └─ Confidence: Depends on model R-squared
│
├─ Application:
│  ├─ Cloud Engineering Demand
│  ├─ Strong correlation with: cloud spending growth (+0.82 correlation)
│  ├─ Forecast: If cloud spending up 15%, demand up ~12%
│  └─ Recommendation: Anticipatory training increase (3-month lag)
│
└─ Advantages:
   ├─ Explains WHY demand changes
   ├─ Actionable insights (can predict based on known trends)
   └─ Good for strategic planning

MODEL 4: TREND ANALYSIS (Lifecycle modeling)
├─ Best for: Identifying emerging/declining skills
├─ Skill Lifecycle Phases:
│  ├─ Emerging (0–24 months):
│  │  ├─ Demand growth rate: 20%+/year
│  │  ├─ Gartner phase: "Peak of Inflated Expectations"
│  │  ├─ Supply response: Low (new, risky)
│  │  └─ Action: First-mover advantage in training
│  │
│  ├─ Growth (24–60 months):
│  │  ├─ Demand growth rate: 10–20%/year
│  │  ├─ Gartner phase: "Trough of Disillusionment" → "Slope of Enlightenment"
│  │  ├─ Supply response: Medium (increasing)
│  │  └─ Action: Expand capacity (catch rising wave)
│  │
│  ├─ Maturity (60+ months):
│  │  ├─ Demand growth rate: 0–5%/year
│  │  ├─ Gartner phase: "Plateau of Productivity"
│  │  ├─ Supply response: High (established, many providers)
│  │  └─ Action: Optimize curriculum (focus on combinations)
│  │
│  └─ Decline (if demand falls):
│     ├─ Demand growth rate: Negative
│     ├─ Technology obsolescence (replaced by newer tech)
│     ├─ Supply response: Slow (practitioners resist leaving)
│     └─ Action: Transition to successor skills
│
├─ Forecasting:
│  ├─ Determine current lifecycle phase
│  ├─ Project phase duration (based on technology patterns)
│  ├─ Forecast demand curve (S-curve model)
│  └─ Identify inflection points (when phase changes occur)
│
└─ Example:
   ├─ Kubernetes: Emerging → Growth transition (2020)
   ├─ Forecast: Growth phase continues through 2026
   ├─ Demand trajectory: 15% annual growth
   └─ Action: Build Kubernetes track capacity

ENSEMBLE FORECASTING (Weighted combination):
├─ Use multiple models, weight by recent performance
├─ Weights based on: Model accuracy (RMSE, MAE) over past 12 months
├─ Final forecast = (40% Exponential + 30% ARIMA + 30% Regression)
├─ Confidence interval: ±10% at 95% CI (for 3-month forecast)
├─ Methodology: Industry best practice (reduces forecast error)
└─ Output: Single forecast with confidence bounds

FORECAST VALIDATION:
├─ Track forecast accuracy quarterly
├─ Compare forecasted demand vs actual (MAE, RMSE)
├─ Update model weights if accuracy declining
├─ Adjust for systematic bias (over/under forecasting)
└─ Transparency: Report forecast error margins monthly
```

```

├─ Interpretation: Skill clearly declining
├─ Action:
│  ├─ Begin capacity reduction (−20% first quarter)

│  └─ Build goodwill for "we helped you pivot" positioning

│  └─ Support extended: Help finding positions in legacy systems

└─ Example: Punch cards, FORTRAN (still used in legacy systems)

│  ├─ Positive gap (shortage) = opportunity

├─ Growth_Trajectory: Phase of technology lifecycle
│  ├─ Emerging (+50): Highest growth, first-mover advantage
│  ├─ Growth (+30): Strong growth, market expansion
│  ├─ Maturity (+10): Stable, competitive
│  ├─ Decline (−30): Shrinking market
│  └─ Obsolete (−50): End of life
│
├─ Strategic_Value: Alignment with company goals
│  ├─ Core (×2): Skills central to platform mission
│  ├─ Complementary (×1.5): Support core skills
│  ├─ Peripheral (×0.8): Nice-to-have
│  └─ Strategic alignment score based on board priorities
│
└─ Example calculation:
   ├─ Kubernetes: Gap=+25, Growth=+30, Strategic=2.0
   ├─ Score = 25 × 30 × 2.0 = 1500 (HIGH PRIORITY)
   └─
   ├─ COBOL: Gap=−10, Growth=−50, Strategic=0.5
   ├─ Score = −10 × (−50) × 0.5 = 250 (LOW PRIORITY)

├─ Total training capacity: Limited by mentors, infrastructure, learners

├─ Constraints:

│  └─ Competitive positioning: What competitors offer

│  ├─ Reserve capacity for high-opportunity skills

│  └─ Buffer capacity (flexibility for surprises)
│
└─ Example budget allocation:
   ├─ Kubernetes: +50% capacity (shortage, growth, strategic)

│  ├─ Scale-up tracks (emerging skills, high demand)
│  ├─ Scale-down tracks (declining skills, low demand)

│  └─ Monitor outcomes (check if forecasts accurate)
│
└─ Phase 3 (Months 10–12): Stabilization
   ├─ Evaluate results (demand match, quality maintained?)
   ├─ Adjust forecasts (update based on actuals)
   ├─ Plan for next period (quarterly cycle)
   └─ Communicate results (transparency to community)

STEP 4: QUALITY GATE CHECKS
├─ Constraint 1: Mentor quality
│  ├─ Don't hire too many mentors too fast (quality dilution)
│  ├─ Training programs for new mentors (3–6 month onboarding)

│  └─ Ramping: Gradual increase in mentor capacity (15–20%/quarter)

│  ├─ Curriculum development time-intensive (8–12 weeks)
│  ├─ Content review & approval gates
│  ├─ Pilot testing before full launch
│  └─ Versioning & iteration (continuous improvement)
│
├─ Constraint 3: Learner experience

│  ├─ Maintain student-mentor ratios (learning quality)
│  ├─ Keep wait times reasonable (learner satisfaction)
│  └─ Monitor NPS & completion rates (leading indicators)
│
└─ Principle: Quality > Quantity (always)

STEP 5: SKILL COMBINATION OPPORTUNITIES
├─ Data analysis: Which skill pairs have high market value?
│  ├─ Python + Machine Learning (highly valued combination)

├─ Calculate: Forecasted demand vs actual demand
├─ Error metric: Mean Absolute Percentage Error (MAPE)
├─ Formula: MAPE = (1/N) × ∑ |Forecast − Actual| / Actual × 100%
├─ Target: <15% MAPE (industry standard)
├─ Alert threshold: >25% MAPE (forecast quality issue)
├─ Actions:
│  ├─ If MAPE increasing: Review data sources + model assumptions
│  ├─ If MAPE acceptable: Continue methodology
│  ├─ If MAPE declining: Investigate possible data error
│  └─ Quarterly review: Adjust model weights/parameters
└─ Transparency: Report MAPE monthly to leadership

METRIC 2: SUPPLY FORECAST ACCURACY
├─ Calculate: Forecasted training completions vs actual
├─ Measurement: Enrollment → Completion rate tracking
├─ Target: Within ±10% of forecast
├─ Actions similar to demand tracking
└─ Challenges:
   ├─ More controllable (internal process)
   ├─ Fewer external variables
   ├─ Higher accuracy expected

METRIC 3: GAP PREDICTION ACCURACY
├─ Definition: Did we correctly predict shortage/surplus?
├─ Measurement: Directional accuracy (shortage vs surplus vs balanced)
├─ Target: >80% directional accuracy
├─ Use: Career advice accuracy (if we said shortage, did shortage occur?)
└─ Implications:
   ├─ Good directional accuracy = learners get accurate career guidance
   ├─ Poor accuracy = learners may be misled about opportunities

METRIC 4: RECOMMENDATION EFFECTIVENESS
├─ Definition: Did curriculum recommendations produce desired outcomes?
├─ Measurement:
│  ├─ If we recommended capacity increase, did placement improve?
│  ├─ If we recommended decrease, did quality improve?
│  ├─ If we recommended obsolescence phase, did attrition match forecast?
│  └─ Outcome vs forecast comparison
│
├─ Success criteria:
│  ├─ Capacity increases: Learner placement rate maintained (>85%)
│  ├─ Capacity decreases: Average salary of graduates increases (>5%)
│  ├─ Obsolescence: Learner transition to successor skill >70%
│  └─ Overall: Curriculum alignment with market outcomes
│
└─ Quarterly review: Did recommendations achieve goals?

METRIC 5: FORECAST STABILITY
├─ Definition: Do forecasts change drastically month-to-month?
├─ Measurement: Month-to-month change in forecast
├─ Target: Smooth changes (not volatile)
├─ Alert threshold: Forecast changes >30 points without explanation
├─ Interpretation:
│  ├─ Volatile forecasts: Lose credibility + user trust
│  ├─ Stable forecasts: Build confidence in system
│  └─ Tool: Smoothing (temper month-to-month volatility)
│
└─ Action:
   ├─ If high volatility: Investigate data quality issues
   ├─ If spikes explained: Document reason + communicate
   └─ If unexplained: Review for modeling errors

├─ Cause: Cannot convert source format to UWDF
├─ Response:

│  ├─ Skip record if unfixable (don't corrupt aggregate)
│  ├─ Alert data engineering team
│  └─ Escalate if >1% of records fail
└─ Example: "Salary data in unexpected format → skip record"

TYPE 3: FORECAST MODEL ERROR
├─ Cause: Mathematical error (division by zero, negative variance)
├─ Response:
│  ├─ Log all inputs + formula state
│  ├─ Halt forecast generation
│  ├─ Use prior month's forecast (conservative fallback)
│  ├─ Alert engineering team
│  └─ Manual review before publishing
└─ Example: "ARIMA calculation overflow → use stable forecast"

TYPE 4: DATABASE WRITE FAILURE
├─ Cause: Cannot save forecast to database
├─ Response:
│  ├─ HALT ALL OPERATIONS

│  ├─ Escalate to database team
