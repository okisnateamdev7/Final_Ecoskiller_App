> the migration so that NO existing record is orphaned, NO existing user is

> a stale or inconsistent label set.

> This agent operates in the **gravity well of all taxonomy change** — it does
> not create taxonomy; it migrates it safely across the entire platform graph.

## 🔄 TAXONOMY MIGRATION LIFECYCLE (SEQUENTIAL — LOCKED)

```
╔══════════════════════════════════════════════════════════════════════╗
║          TAXONOMY MIGRATION LIFECYCLE — 10 GATES                    ║
║          (All gates are mandatory. No gate may be skipped.)          ║
╚══════════════════════════════════════════════════════════════════════╝

GATE 0 → INTAKE & CLASSIFICATION
  ├── Validate trigger source (TXN-TRG-001 to TXN-TRG-015)
  ├── Identify TAXONOMY DOMAIN affected (from governed list above)
  ├── Classify change type:
  │     ADD       → New node added to taxonomy tree
  │     RENAME    → Existing node label changed
  │     MERGE     → Two nodes collapsed into one canonical node
  │     SPLIT     → One node divided into two or more child nodes
  │     DEPRECATE → Node retired; mapped to successor
  │     REORDER   → Hierarchy restructured (parent-child changes)
  │     RE-WEIGHT → Scoring/rubric weights modified (belt/assessment only)
  │     DELETE    → Node permanently removed (REQUIRES P0 approval)
  └── Reject if change type undetermined → LOG + STOP

GATE 1 → IMPACT ASSESSMENT
  ├── Query all entities referencing the affected taxonomy node(s):
  │     - student profiles (skill tags, domain track, course enrollment)
  │     - trainer profiles (skill tags, domain specialization)
  │     - job postings (job categories, required skills)
  │     - belt records (rubric version, proficiency level)
  │     - certifications (credential type, issuer classification)
  │     - dojo sessions (scenario taxonomy, difficulty band)
  │     - learning paths (module → skill alignment)
  │     - AI model label sets (skill vectors, classification labels)
  │     - analytics dashboards (Skill_Demand_Stats, Job_Velocity_Stats)
  ├── Produce: IMPACT_MANIFEST (count of affected records per entity type)
  └── If IMPACT_MANIFEST shows > 10,000 records → escalate to P1 migration

GATE 2 → DEPENDENCY GRAPH RESOLUTION
  ├── Resolve full dependency graph for affected taxonomy node:
  │     Direct dependencies  → entities directly referencing node
  │     Indirect dependencies → entities referencing dependent nodes
  │     AI model dependencies → label sets, training data references
  │     Elasticsearch index dependencies → indexed taxonomy values
  │     Compliance audit dependencies → taxonomy values in audit logs
  ├── Map circular dependency risks (a node depending on its own successor)
  └── Circular dependency detected → QUARANTINE + ESCALATE

GATE 3 → MIGRATION PLAN GENERATION
  ├── Generate MIGRATION_PLAN document containing:
  │     migration_id         : UUID v4
  │     taxonomy_domain      : one of the 15 governed domains
  │     change_type          : (from GATE 0 classification)
  │     affected_nodes       : list of taxonomy node IDs
  │     successor_mapping    : {old_node_id → new_node_id} (for RENAME/MERGE/DEPRECATE)
  │     impact_manifest      : output from GATE 1
  │     dependency_graph     : output from GATE 2
  │     rollback_snapshot_id : UUID of pre-migration state snapshot
  │     migration_strategy   : ZERO_DOWNTIME | SCHEDULED_MAINTENANCE
  │     estimated_duration   : seconds (based on record count)
  │     human_approval_required : true/false (always true for DELETE, RE-WEIGHT, SPLIT)
  ├── Migration plan stored in: taxonomy_migration_plans table
  └── Emit Kafka: taxonomy.migration.plan.ready

GATE 4 → HUMAN APPROVAL CHECKPOINT
  ├── ALWAYS required for: DELETE, RE-WEIGHT, SPLIT, MERGE > 5,000 records
  ├── Required for: Any belt rubric change (BELT_RUBRIC_TAXONOMY)
  ├── Required for: Any data classification sensitivity change (DATA_CLASSIFICATION_TREE)
  ├── Required for: P1/P0 migrations (high impact)
  ├── Approval roles: Admin Governance + Compliance Admin (2-of-3 sign-off)
  ├── Approval window: 72 hours max (P0: 4 hours)
  ├── If rejected → migration CANCELLED; reason logged; taxonomy unchanged
  └── If approved → proceed to GATE 5; approval recorded in audit log

GATE 5 → PRE-MIGRATION SNAPSHOT
  ├── Capture full taxonomy state snapshot (all 15 domains)
  ├── Store in: taxonomy_snapshots table
  ├── Generate SHA-256 hash of snapshot
  ├── Write snapshot hash to compliance_record (immutable)
  ├── Elasticsearch: capture current index mapping for affected domain
  └── snapshot_id referenced in MIGRATION_PLAN and AUDIT_LOG

GATE 6 → MIGRATION EXECUTION (ZERO-DOWNTIME WHERE POSSIBLE)
  ├── Strategy: ZERO_DOWNTIME (preferred)
  │     Phase A: Write new taxonomy node alongside old (dual-write)
  │     Phase B: Backfill all entity references (batched, async)
  │     Phase C: Redirect all new writes to new node
  │     Phase D: Validate zero references remain on old node
  │     Phase E: Deprecate old node (mark, do not delete)
  │
  ├── Strategy: SCHEDULED_MAINTENANCE (only for full-tree restructure)
  │     Announce maintenance window via Notification Service
  │     Execute migration in maintenance window
  │     Re-enable services post-validation
  │
  ├── Batch Processing Rules:
  │     Max batch size: 500 records per transaction
  │     Inter-batch delay: 100ms (prevent DB lock contention)
  │     Failed batch: rollback batch → retry 3x → quarantine + alert
  │
  ├── Elasticsearch Re-indexing:
  │     Update index mapping for taxonomy domain
  │     Re-index all affected documents
  │     Validate index consistency before marking complete
  │
  └── AI Model Label Sync:
        Emit: taxonomy.labels.updated to AI agent event bus
        All AI agents MUST re-pull label set before next inference run
        AI inference BLOCKED on stale taxonomy label sets

GATE 7 → POST-MIGRATION VALIDATION
  ├── Integrity checks:
  │     Zero orphaned entity references to deprecated nodes
  │     Zero broken FK references in normalized_records
  │     Belt records: no rubric version mismatch
  │     Certification records: credential type maps to valid taxonomy node
  │     Analytics: Skill_Demand_Stats reflects new taxonomy structure
  ├── AI Label Validation:
  │     All AI agents confirm receipt of updated label set
  │     Confidence score distribution checked for anomalies post-migration
  ├── Elasticsearch:
  │     Search result accuracy test against new taxonomy structure
  ├── Parent Portal:
  │     Read-only snapshots reflect updated taxonomy labels
  └── Validation failure → ROLLBACK (GATE 8)

GATE 8 → ROLLBACK PROTOCOL (TRIGGERED ONLY ON VALIDATION FAILURE)
  ├── Restore from snapshot captured at GATE 5
  ├── Restore Elasticsearch index from pre-migration snapshot
  ├── Emit: taxonomy.migration.rolled_back
  ├── Notify Admin Governance + Compliance Admin
  ├── Log full rollback report to normalization_audit_log
  └── Migration marked: STATUS = ROLLED_BACK (non-deletable record)

GATE 9 → SEAL & PUBLISH
  ├── Increment taxonomy version: SEMVER (MAJOR.MINOR.PATCH)
  │     ADD / RENAME / REORDER / RE-WEIGHT → MINOR or PATCH
  │     MERGE / SPLIT / DEPRECATE → MINOR
  │     DELETE (full node removal) → MAJOR
  ├── Publish new taxonomy version to:
  │     taxonomy_version_registry (authoritative record)
  │     Kafka: taxonomy.version.published
  │     Elasticsearch: taxonomy index metadata updated
  │     AI agent label feed: taxonomy.labels.v{VERSION}
  │     Admin Governance dashboard
  │     Institute ERP taxonomy reference cache
  ├── Update taxonomy_audit_log (append-only, immutable)
  └── Close migration: STATUS = COMPLETED · SEALED

GATE 10 → DRIFT MONITORING ACTIVATION
  ├── Schedule post-migration drift scan: +24h, +7d, +30d
  ├── Drift scan checks:
  │     New entity records using deprecated taxonomy nodes
  │     AI inference outputs using old label values
  │     Analytics aggregations referencing stale taxonomy keys
  ├── Drift detected → auto-trigger: TXN-TRG-009
  └── Drift scan results logged to taxonomy_drift_log
```

## 📐 TAXONOMY GOVERNANCE RULES (SEALED — IMMUTABLE)

### RULE-T01 | Taxonomy Versioning (MANDATORY)
```
Every taxonomy is versioned using SEMANTIC VERSIONING (MAJOR.MINOR.PATCH).
Every migration increments the version.
Version history is IMMUTABLE — no version may be deleted.
Rollbacks restore state to previous version; they do NOT delete the failed version.
Version N and Version N-1 MUST coexist in the registry during migration windows.
```

### RULE-T02 | No Silent Deprecation
```
No taxonomy node may be deprecated without:
  1. A recorded successor_mapping (old_node_id → new_node_id)
  2. A published migration plan
  3. Admin approval (if required by change type)
  4. All entity references updated or flagged
  5. A migration completion seal in taxonomy_audit_log

Silent deprecation = INTEGRITY FAILURE → STOP EXECUTION + P0 ALERT
```

### RULE-T03 | Belt & Certification Integrity Lock
```
Belt rubric changes (RE-WEIGHT, SPLIT, MERGE in BELT_RUBRIC_TAXONOMY) REQUIRE:
  1. Mentor board review (human — not AI)
  2. 2-of-3 Admin Governance sign-off
  3. Re-certification trigger evaluation:
       Major rubric change (MAJOR version bump) → re-certification required for all
       existing belt holders at affected belt level.
  4. No student's belt rank may be silently downgraded by a taxonomy migration.
  5. Belt holders affected by deprecation → notified via Notification Service.
  
AI CANNOT directly award belts. Mentor confirmation is mandatory.
Taxonomy migration CANNOT alter existing awarded belt records retroactively
without explicit human approval and audit trail.
```

### RULE-T04 | 300-Role Taxonomy Integrity
```
The 300-role taxonomy is the authoritative role classification for all
Ecoskiller users. Migrations to ROLE_TAXONOMY must:
  1. Maintain full coverage: no user may be left without a mapped role.
  2. Maintain domain isolation: role changes cannot merge across
     Arts | Commerce | Science | Technology | Administration.
  3. RBAC + ABAC permission inheritance must be re-evaluated after
     any role taxonomy migration.
  4. All user sessions must be invalidated and re-authenticated
     following a MAJOR ROLE_TAXONOMY migration.
  5. Role taxonomy migration completion must be verified by
     Compliance Admin before platform resumes role-gated operations.
```

### RULE-T05 | Cross-Domain Taxonomy Isolation
```
DOMAIN_TRACKS = Arts | Commerce | Science | Technology | Administration

Taxonomy nodes are domain-scoped. A skill node in Technology domain
CANNOT be migrated to Commerce domain without:
  1. Full impact assessment across all entities
  2. Explicit cross-domain migration approval (Super Admin + Compliance Admin)
  3. Domain isolation verification post-migration

Cross-domain taxonomy bleed = SECURITY FAILURE → STOP EXECUTION
```

### RULE-T06 | AI Model Label Contract
```
AI models (resume parser, skill gap detector, match scorer, placement predictor,
recruiter behavior AI) consume taxonomy labels as their classification vocabulary.

On ANY taxonomy migration:
  1. AI agents MUST be notified via Kafka: taxonomy.labels.updated
  2. AI inference is BLOCKED until agent confirms label receipt
  3. AI models MUST NOT run inference on deprecated taxonomy labels
  4. Label set for each AI model stored as: ai_label_contract_v{VERSION}
  5. Confidence score drift > 10% post-migration → trigger TXN-TRG-010
  6. AI label contracts are versioned and immutable in the registry
```

### RULE-T07 | Elasticsearch Index Governance
```
Taxonomy drives Elasticsearch index structure for:
  - Skill search
  - Job search
  - Trainer discovery
  - Student profile search
  - Course catalog search

On taxonomy migration:
  1. Index mapping MUST be updated (not recreated unless MAJOR version)
  2. Zero-downtime re-indexing required (blue-green index pattern)
  3. Search accuracy must be validated post-migration
  4. Deprecated taxonomy values MUST remain searchable for 90 days
     (via alias mapping) to maintain SEO continuity on React web
  5. Elasticsearch migration is ATOMIC with entity migration —
     they CANNOT diverge.
```

### RULE-T08 | Data Classification Taxonomy (L0–L4) Governance
```
DATA_CLASSIFICATION_TREE governs sensitivity labels L0 (public) through L4 (critical).

Changes to data classification taxonomy require:
  1. Compliance Admin approval (mandatory — no exceptions)
  2. DLP policy update synchronized with taxonomy change
  3. Re-labeling of all affected data assets
  4. Audit report generated for regulatory compliance
  5. Annual taxonomy refresh (per Section 162.16) is a mandatory cycle
     — agent MUST initiate and complete this cycle on schedule.

Failure to complete Annual Taxonomy Refresh → COMPLIANCE FAILURE → ESCALATE
```

### RULE-T09 | Tenant Isolation in Taxonomy Migration
```
Taxonomy migrations are PLATFORM-WIDE (master taxonomy) or
TENANT-SCOPED (institute-specific curriculum maps, custom skill tags).

Platform-wide migration affects all tenants simultaneously.
Tenant-scoped migration affects only the requesting tenant's scope.

Cross-tenant taxonomy contamination = SECURITY FAILURE → STOP EXECUTION

Tenant-scoped custom taxonomy nodes MUST:
  - Carry tenant_id on every node
  - Not conflict with platform master taxonomy namespaces
  - Be prefixed: custom.{tenant_id}.{node_name}
```

### RULE-T10 | Curriculum & Learning Path Integrity
```
COURSE_SUBJECT_MAP and LEARNING_PATH_MAP migrations must:
  1. Preserve all active student enrollment records
  2. Map deprecated subjects to successor subjects
  3. Re-evaluate learning path completions against new structure:
       Student with 80% on deprecated path → calculate completion equivalence
       on new path; NEVER retroactively fail a completed path.
  4. Institute admins notified of curriculum taxonomy migration completion.
  5. Learning objectives mapped to skills must remain traceable after migration.
```

### RULE-T11 | Dojo Scenario Taxonomy Integrity
```
DOJO_SCENARIO_TAXONOMY changes must not:
  - Invalidate historic match results
  - Change difficulty band of completed sessions retroactively
  - Alter scoring rubric for completed and sealed matches

All future sessions use new taxonomy.
All historic sessions remain sealed under the taxonomy version active at time of play.
```

### RULE-T12 | Technology Stack Map Governance
```
TECHNOLOGY_STACK_MAP tracks all technologies, frameworks, languages, and tools
referenced on the platform (in jobs, skills, courses, projects, dojo scenarios).

New technologies are added with:
  - canonical_name (normalized, lowercase-hyphenated)
  - category (language | framework | library | tool | platform | cloud | database)
  - domain_track (Technology preferred; may span multiple)
  - version_awareness flag (true if version matters for skill matching)
  - deprecation_status (active | sunset | legacy | archived)

Technology nodes set to archived MUST remain in taxonomy for historic record integrity.
```

## 🗄️ DATABASE SCHEMA (POSTGRESQL — INSTITUTE ERP)

```sql
-- Taxonomy Version Registry (Master Source of Truth)
CREATE TABLE taxonomy_version_registry (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  taxonomy_domain       VARCHAR(60)  NOT NULL,
  version_major         INT          NOT NULL DEFAULT 1,
  version_minor         INT          NOT NULL DEFAULT 0,
  version_patch         INT          NOT NULL DEFAULT 0,
  version_string        VARCHAR(20)  GENERATED ALWAYS AS
                          (version_major||'.'||version_minor||'.'||version_patch) STORED,
  status                VARCHAR(20)  NOT NULL DEFAULT 'ACTIVE',
  published_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
  published_by_agent    VARCHAR(100) NOT NULL DEFAULT 'taxonomy_migration_agent',
  approved_by           UUID[]       REFERENCES users(id),
  snapshot_id           UUID,
  CONSTRAINT chk_taxonomy_domain CHECK (taxonomy_domain IN (
    'SKILL_TAXONOMY','ROLE_TAXONOMY','JOB_TAXONOMY','DOMAIN_TRACK_REGISTRY',
    'BELT_RUBRIC_TAXONOMY','COMPETENCY_FRAMEWORK','COURSE_SUBJECT_MAP',
    'CERTIFICATION_TAXONOMY','INDUSTRY_DEMAND_MAP','BADGE_CRITERIA_TAXONOMY',
    'TECHNOLOGY_STACK_MAP','DOJO_SCENARIO_TAXONOMY','EVIDENCE_TAXONOMY',
    'DATA_CLASSIFICATION_TREE','LEARNING_PATH_MAP'
  )),
  CONSTRAINT chk_version_status CHECK (
    status IN ('ACTIVE','DEPRECATED','ROLLED_BACK','ARCHIVED')
  )
);

-- Taxonomy Nodes (Universal Node Registry)
CREATE TABLE taxonomy_nodes (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  taxonomy_domain       VARCHAR(60)  NOT NULL,
  node_code             VARCHAR(100) NOT NULL,
  canonical_name        VARCHAR(255) NOT NULL,
  display_name          VARCHAR(255),
  parent_node_id        UUID         REFERENCES taxonomy_nodes(id),
  domain_track          VARCHAR(30),
  tenant_id             UUID,        -- NULL = platform-wide node
  node_status           VARCHAR(20)  NOT NULL DEFAULT 'ACTIVE',
  successor_node_id     UUID         REFERENCES taxonomy_nodes(id),
  taxonomy_version_id   UUID         NOT NULL REFERENCES taxonomy_version_registry(id),
  metadata              JSONB,
  created_at            TIMESTAMPTZ  NOT NULL DEFAULT now(),
  deprecated_at         TIMESTAMPTZ,
  CONSTRAINT chk_node_status CHECK (
    node_status IN ('ACTIVE','DEPRECATED','MERGED','SPLIT','ARCHIVED')
  ),
  UNIQUE(taxonomy_domain, node_code, taxonomy_version_id)
);

-- Taxonomy Migration Plans
CREATE TABLE taxonomy_migration_plans (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  migration_id          UUID         UNIQUE NOT NULL DEFAULT gen_random_uuid(),
  taxonomy_domain       VARCHAR(60)  NOT NULL,
  trigger_id            VARCHAR(20)  NOT NULL,
  change_type           VARCHAR(20)  NOT NULL,
  affected_node_ids     UUID[]       NOT NULL,
  successor_mapping     JSONB,
  impact_manifest       JSONB        NOT NULL,
  dependency_graph      JSONB        NOT NULL,
  rollback_snapshot_id  UUID,
  migration_strategy    VARCHAR(30)  NOT NULL DEFAULT 'ZERO_DOWNTIME',
  estimated_duration_s  INT,
  human_approval_required BOOLEAN   NOT NULL DEFAULT TRUE,
  approved_by           UUID[],
  approved_at           TIMESTAMPTZ,
  status                VARCHAR(30)  NOT NULL DEFAULT 'PENDING_APPROVAL',
  started_at            TIMESTAMPTZ,
  completed_at          TIMESTAMPTZ,
  created_at            TIMESTAMPTZ  NOT NULL DEFAULT now(),
  CONSTRAINT chk_change_type CHECK (change_type IN (
    'ADD','RENAME','MERGE','SPLIT','DEPRECATE','REORDER','RE-WEIGHT','DELETE'
  )),
  CONSTRAINT chk_plan_status CHECK (status IN (
    'PENDING_APPROVAL','APPROVED','REJECTED','IN_PROGRESS',
    'COMPLETED','ROLLED_BACK','CANCELLED','FAILED'
  ))
);

-- Taxonomy Snapshots (Pre-migration full state captures)
CREATE TABLE taxonomy_snapshots (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  snapshot_hash         VARCHAR(64)  NOT NULL UNIQUE,  -- SHA-256
  taxonomy_domain       VARCHAR(60),  -- NULL = full platform snapshot

  migration_plan_id     UUID         REFERENCES taxonomy_migration_plans(id),

-- AI Label Contracts (Versioned per taxonomy version)
CREATE TABLE ai_label_contracts (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  label_set             JSONB        NOT NULL,
  contract_hash         VARCHAR(64)  NOT NULL,  -- SHA-256 of label_set

  consumed_by_agents    JSONB,        -- {agent_id: confirmed_at}
  UNIQUE(taxonomy_domain, taxonomy_version_id)

-- Taxonomy Drift Log
CREATE TABLE taxonomy_drift_log (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scan_timestamp        TIMESTAMPTZ  NOT NULL DEFAULT now(),

  affected_record_ids   UUID[],
  drift_detail          JSONB        NOT NULL,
  auto_trigger_id       VARCHAR(20),
  resolved_at           TIMESTAMPTZ,
  resolved_by           UUID         REFERENCES users(id)

-- Taxonomy Audit Log (APPEND-ONLY — NO UPDATE, NO DELETE)
CREATE TABLE taxonomy_audit_log (

  taxonomy_version_id   UUID,

  actor_human           UUID         REFERENCES users(id),
  outcome               VARCHAR(20)  NOT NULL,
  detail                JSONB,
  timestamp_utc         TIMESTAMPTZ  NOT NULL DEFAULT now()
  -- ROW LEVEL SECURITY: INSERT ONLY
);
```
