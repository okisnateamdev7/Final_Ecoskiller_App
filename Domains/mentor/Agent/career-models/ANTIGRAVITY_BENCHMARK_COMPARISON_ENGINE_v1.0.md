# 🔒 SECTION 2 — WHAT THIS MODULE DOES (SYSTEM DEFINITION)

**ANTIGRAVITY Benchmark Comparison Engine** is the ML-powered benchmark intelligence layer — the fourth and integrative module in the ANTIGRAVITY system. It consumes the full output of all three prior ANTIGRAVITY modules and transforms raw performance data into governed, statistically rigorous, contextually fair, multi-dimensional benchmark standards that every institute can compare itself against — nationally, regionally, by domain, by cohort size, by industry sector alignment, and against HR recruiter expectations.

Its singular operational mandate is:

> **Construct, maintain, publish, and govern the complete benchmark intelligence system for every institute on the ECOSKILLER platform — delivering fair, statistically normalized, contextually segmented comparison intelligence across eight benchmark planes, with full anonymization of peer identities, full HR and recruiter demand alignment, and full longitudinal benchmark evolution tracking — so that every institute always knows exactly where it stands, why it stands there, and what the gap to the next benchmark tier actually costs.**

This module answers seven core benchmark intelligence questions:

```
QUESTION 1 — WHAT IS THE NATIONAL STANDARD FOR MY INSTITUTE TYPE?
  Fair, domain-normalized, tier-normalized benchmark standards
  computed from all comparable institutes on the platform.

QUESTION 2 — WHERE DO I RANK AMONG GENUINE PEERS?
  Propensity-score matched peer cluster assignment ensuring
  only truly comparable institutes form each cluster —
  same domain, same size band, same tier, same region band.

QUESTION 3 — HOW WIDE IS MY GAP TO THE NEXT BENCHMARK TIER?
  Precise, dimension-by-dimension gap quantification showing
  exactly how many points and which actions separate an
  institute from the next benchmark classification.

QUESTION 4 — WHAT DO HR RECRUITERS ACTUALLY EXPECT FROM MY GRADUATES?
  A live HR expectation benchmark derived from recruiter
  demand signals, placement outcome data, and employer
  satisfaction scores — not self-declared standards.

QUESTION 5 — HOW AM I TRENDING AGAINST BENCHMARKS OVER TIME?
  Longitudinal benchmark tracking showing whether an institute
  is converging toward or diverging from benchmark standards
  over rolling 30d / 90d / 180d / 12-month windows.

QUESTION 6 — WHICH BENCHMARK GAPS COST THE MOST REVENUE?
  A benchmark-to-revenue impact analysis that quantifies the
  financial cost of every benchmark gap using signals from
  the ANTIGRAVITY Revenue Forecast Model.

QUESTION 7 — WHERE DOES MY REGION STAND AND WHAT DOES THAT MEAN FOR ME?
  Regional benchmark intelligence showing how an institute
  performs relative to regional peers and national averages —
  with region-specific HR demand context.
```

# 🔒 SECTION 3 — BENCHMARK ARCHITECTURE: EIGHT BENCHMARK PLANES

```
BENCHMARK PLANE 1 — NATIONAL STANDARD BENCHMARK
  Definition : Platform-wide performance standard computed from all
               active institutes, normalized by domain and institute type.
  Computed   : Weekly (full recompute) + triggered on major data events
  Output     : national_mean, national_p25, national_p50, national_p75,
               national_p90 per dimension + composite
  Exposure   : Institutes see their position vs distribution.
               No individual peer identity exposed.

BENCHMARK PLANE 2 — DOMAIN BENCHMARK
  Definition : Standard computed only among institutes operating in
               the same declared domain track(s).
  Domains    : Arts | Commerce | Science | Technology | Administration
  Multi-domain institutes scored against each active domain separately.
  Output     : domain_mean, domain_p25, domain_p50, domain_p75, domain_p90
               per dimension + composite, per active domain

BENCHMARK PLANE 3 — PEER CLUSTER BENCHMARK
  Definition : Tightest comparison layer. Computed among institutes
               matched on: domain + cohort size band + institute tier type
               + region band + platform tenure band.
  Method     : Propensity Score Matching (PSM) + K-Means clustering
               re-run monthly.
  Output     : cluster_mean, cluster_p25, cluster_p50, cluster_p75, cluster_p90
               per dimension + composite
               cluster_size (number of institutes in cluster)
               cluster_match_quality_score (0.0–1.0)
  Guarantee  : Minimum 5 institutes per cluster. If < 5, cluster merged
               with nearest domain cluster. No singleton clusters allowed.

BENCHMARK PLANE 4 — HR EXPECTATION BENCHMARK
  Definition : What recruiters and employers actually demand from graduates
               of institute type in this domain — derived from live
               placement outcome data, employer satisfaction signals,
               recruiter behavior analytics, and job demand signals.
  Source     : Job Portal Engine + HR Integration Engine + ANTIGRAVITY
               Gap Detector (Plane 2) + Revenue Model (Plane 2)
  Output     : hr_expectation_score per skill dimension
               hr_satisfaction_threshold (minimum employer satisfaction)
               hr_placement_rate_expectation
               hr_ctc_benchmark_ratio
               hr_skill_readiness_threshold
               hr_benchmark_gap (institute score vs HR expectation)

BENCHMARK PLANE 5 — REGIONAL BENCHMARK
  Definition : Standard computed among institutes in the same geographic
               region band — enabling fair regional context.
  Regions    : Platform-configurable regional taxonomy (state / province /
               metro area / national — per deployment region)
  Output     : regional_mean, regional_p25, regional_p50, regional_p75,
               regional_p90 per dimension + composite
               regional_rank (institute position within region)
               region_vs_national_delta (regional standard vs national)

BENCHMARK PLANE 6 — COHORT SIZE BENCHMARK
  Definition : Comparison among institutes of similar student cohort sizes —
               ensuring large institutes are not benchmarked unfairly
               against small ones and vice versa.
  Size Bands : MICRO (1–50 students) | SMALL (51–200) | MEDIUM (201–1000)
               LARGE (1001–5000) | ENTERPRISE (5001+)
  Output     : size_band_mean, size_band_p50, size_band_p75 per dimension
               size_band_rank

BENCHMARK PLANE 7 — BELT & CERTIFICATION BENCHMARK
  Definition : Dojo-specific benchmark measuring belt progression velocity,
               belt quality (certification fee legitimacy), belt validity
               correlation, and belt-to-placement conversion rate — compared
               against domain peers.
  Source     : Belt Engine + ANTIGRAVITY Gap Detector (Belt Validity) +
               Revenue Model (Belt Revenue Plane)
  Output     : belt_velocity_benchmark_p50 / p75 / p90
               belt_quality_score_benchmark
               belt_validity_correlation_benchmark
               belt_to_placement_conversion_benchmark
               belt_governance_compliance_benchmark

BENCHMARK PLANE 8 — CURRICULUM QUALITY BENCHMARK
  Definition : Curriculum-specific benchmark comparing curriculum health
               scores, phantom skill rates, rubric integrity compliance,
               scenario calibration accuracy, and market alignment scores
               against platform-wide standards.
  Source     : ANTIGRAVITY Gap Detector v1.0 (ALL signals — direct feed)
  Output     : curriculum_health_score_benchmark_p50 / p75 / p90
               phantom_skill_rate_benchmark
               rubric_integrity_compliance_rate_benchmark
               market_alignment_score_benchmark
               scenario_accuracy_benchmark
```

# 🔒 SECTION 5 — ML MODEL ARCHITECTURE (LOCKED)

```
MODEL_B1: PEER_CLUSTER_FORMATION_ENGINE
  Type          : Hierarchical Clustering + Propensity Score Matching (PSM)
                  Two-stage: PSM for initial propensity scoring →
                  K-Means on propensity scores for cluster formation
  Matching Vars : domain_tracks · cohort_size_band · institute_type ·
                  region_band · platform_tenure_band · subscription_tier
  Output        : cluster_id per institute
                  cluster_assignment_score (0.0–1.0 quality)
                  cluster_size
                  cluster_centroid_profile
                  nearest_alternative_cluster_id
                  cluster_stability_score (how stable assignment is over time)
  Constraints   : Minimum 5 institutes per cluster (HARD)
                  Maximum intra-cluster variance threshold enforced
                  Single-institute clusters FORBIDDEN
  Trigger       : Run monthly (full re-clustering)
                  Run on significant institute_segmentation_profile change
  Retrain       : Monthly

MODEL_B2: NATIONAL_BENCHMARK_STANDARD_COMPUTER
  Type          : Robust Statistical Estimation
                  (Winsorized mean + Bootstrap confidence intervals +
                   Kernel Density Estimation for distribution fitting)
  Input         : All institute performance payloads +
                  domain segmentation + region segmentation +
                  cohort size segmentation
  Output per benchmark plane per segment:
                  mean, std_dev
                  p10, p25, p50, p75, p90, p95
                  confidence_interval_95 per percentile
                  distribution_shape (NORMAL | SKEWED_LEFT | SKEWED_RIGHT | BIMODAL)
                  outlier_flag_threshold (3 std dev)
                  sample_size_institutes
  Trigger       : Run weekly (full recompute)
  Retrain       : N/A (statistical computation — no ML training)
  Validation    : Kolmogorov-Smirnov test on distribution each run

MODEL_B3: BENCHMARK_TIER_CLASSIFIER
  Type          : Multi-class Classification
                  (Random Forest + Isotonic Calibration)
  Input         : Institute scores per benchmark plane +
                  benchmark_standard_version (current percentile thresholds) +
                  institute_segmentation_profile
  Output        : benchmark_tier per plane:
                    BENCHMARK_LEADER      (≥ p90)
                    BENCHMARK_STRONG      (p75 – p89)
                    BENCHMARK_STANDARD    (p50 – p74)
                    BENCHMARK_DEVELOPING  (p25 – p49)
                    BENCHMARK_LAGGING     (< p25)
                  tier_confidence (0.0–1.0)
                  tier_margin               -- distance to next tier boundary
                  tier_boundary_score       -- exact score needed for next tier
                  tier_direction_30d        -- CONVERGING | STABLE | DIVERGING
  Trigger       : Run on every new performance payload received per institute
  Retrain       : Monthly

MODEL_B4: BENCHMARK_GAP_QUANTIFIER
  Type          : Regression + Quantile Regression
                  (XGBoost + Pinball Loss for quantile estimation)
  Input         : Institute score per dimension per benchmark plane +
                  current benchmark standard (all percentiles) +
                  peer cluster statistics
  Output per dimension per benchmark plane:
                  gap_to_p50 (points)
                  gap_to_p75 (points)
                  gap_to_p90 (points)
                  gap_to_peer_cluster_mean (points)
                  gap_direction (ABOVE | AT | BELOW)
                  gap_severity (CRITICAL | HIGH | MEDIUM | LOW | NONE)
                  projected_gap_closure_days (if below — based on trend)
                  projected_gap_widening_days (if above — trend risk)
  Trigger       : Run on every new performance payload received
  Retrain       : Monthly

MODEL_B5: HR_EXPECTATION_BENCHMARK_ENGINE
  Type          : Bayesian Hierarchical Model
                  (Institute type → domain → region hierarchy)
  Input         : RECRUITER_DEMAND_BENCHMARK_SIGNAL +
                  EMPLOYER_SATISFACTION_BENCHMARK_LOG +
                  PLACEMENT_BENCHMARK_AGGREGATE +
                  market_demand_signals (from ANTIGRAVITY Gap Detector Plane 2)
  Output        : hr_expectation_composite_score per domain per region
                  per skill dimension:
                    hr_minimum_threshold
                    hr_target_threshold
                    hr_elite_threshold
                  hr_gap per institute (score vs HR minimum/target/elite)
                  hr_benchmark_tier:
                    HR_EXCEEDS       (≥ HR elite threshold)
                    HR_MEETS         (≥ HR target threshold)
                    HR_APPROACHING   (≥ HR minimum threshold)
                    HR_BELOW         (< HR minimum threshold)
  Trigger       : Run weekly + on significant recruiter_demand_signal change
  Retrain       : Monthly (posterior update with new employer data)

MODEL_B6: BENCHMARK_REVENUE_IMPACT_QUANTIFIER

  Output        : cluster_stability_score_platform (0.0–1.0)

# 🔒 SECTION 7 — BENCHMARK RULE ENGINE (IMMUTABLE)

```
RULE_B001 — MINIMUM CLUSTER SIZE RULE
  A benchmark cluster MUST contain minimum 5 institutes.
  IF proposed cluster < 5 institutes:
    THEN cluster MERGED with nearest domain cluster
         merge_reason logged
         affected institutes notified
  Singleton clusters: FORBIDDEN
  Clusters of 2–4: FORBIDDEN
  Violation → STOP BENCHMARK COMPUTATION · REPORT CLUSTER_SIZE_VIOLATION

RULE_B002 — BENCHMARK STANDARD MINIMUM SAMPLE RULE
  A benchmark standard MAY NOT be published with fewer than
  10 institutes in the computation sample.
  IF sample_size < 10:
    THEN standard_validity_score = INSUFFICIENT_SAMPLE
         benchmark displayed as PRELIMINARY (not authoritative)
         Governance Board notification required
         public publication of this standard = FORBIDDEN

RULE_B003 — BENCHMARK STANDARD VERSION GOVERNANCE RULE
  Any change to benchmark standard methodology, computation method,
  or tier threshold requires:
    - Governance Board approval
    - Version bump in benchmark_standard_version_log
    - 30-day notice period before new standard takes effect
    - All institutes notified of upcoming change
    - Historic standards preserved in version log (immutable)
  Silent benchmark standard changes: FORBIDDEN
  Retroactive benchmark reclassification: FORBIDDEN

RULE_B004 — PEER ANONYMITY ABSOLUTE RULE
  Under no circumstance may the identity of any peer institute be
  revealed through any benchmark output, API response, UI screen,
  exported report, notification, or computed value.
  Cluster IDs are internal UUIDs — never exposed externally.
  Cluster descriptions use ONLY statistical parameters.
  Cluster size ROUNDED to nearest 5 in all public/institute-facing outputs.
  Violation → STOP EXECUTION · SECURITY INCIDENT · GOVERNANCE ESCALATION

RULE_B005 — BENCHMARK TIER LAGGING ESCALATION RULE
  IF institute benchmark_tier = BENCHMARK_LAGGING (< p25) on
  NATIONAL STANDARD BENCHMARK (Plane 1) for > 60 consecutive days:
    THEN Governance Board review automatically scheduled
         EWS signal sent to Performance Predictor
         Revenue impact report generated
         platform_admin_alert = IMMEDIATE
  IF < p10 for > 30 consecutive days:
    THEN public_trust_badge = UNDER_REVIEW
         talent_marketplace_visibility = ADVISORY_ONLY

RULE_B006 — HR BENCHMARK BELOW ESCALATION RULE
  IF hr_benchmark_tier = HR_BELOW for any active domain:
    THEN placement_revenue_risk_signal sent to Revenue Model
         Gap Detector notified (market alignment gap confirmed)
         Institute admin notified within 24 hours
         HR gap closure plan RECOMMENDED (advisory)
  IF HR_BELOW persists > 90 days:
    THEN governance_board_review scheduled
         public employer-facing badge degraded

RULE_B007 — BENCHMARK DIVERGENCE ALERT RULE
  IF benchmark_convergence_label = FAST_DIVERGING (≥ 2.0 pts/month away
  from national p50 benchmark):
    THEN divergence_alert = True
         performance_predictor EWS integration signal sent
         revenue_model churn risk signal checked
         institute admin notified within 24 hours
         improvement advisory issued automatically
         (advisory only — no auto-penalty)

RULE_B008 — BENCHMARK REVENUE IMPACT MINIMUM DISPLAY RULE
  IF revenue_cost_of_gap > platform_configured_disclosure_threshold:
    THEN revenue impact MUST be displayed alongside benchmark gap
         in all dashboard surfaces and exported reports
         Institute admin MUST see revenue cost of every CRITICAL gap
  Revenue impact is advisory context — not a fine or penalty.
  Platform_configured_disclosure_threshold set by Platform Admin.

RULE_B009 — BENCHMARK CLUSTER DRIFT NOTIFICATION RULE
  IF an institute's cluster assignment changes between monthly runs:
    THEN institute_admin notified of cluster reassignment
         reassignment_reason provided (statistical reason only)
         new cluster benchmark statistics provided
         Previous cluster benchmark history preserved (not deleted)
  Cluster drift does NOT retroactively reclassify historical tiers.

RULE_B010 — BENCHMARK DISPUTE RESPONSE RULE
  IF an institute raises a benchmark dispute:
    THEN Governance Board must acknowledge within 5 business days
         Governance Board must resolve within 30 calendar days
  IF no resolution within 30 days:
    THEN dispute automatically escalated to Platform Admin
         benchmark_dispute_log updated with escalation event
  Resolution options:
    UPHOLD_ORIGINAL   : No change — original computation confirmed correct
    RECLUSTER         : Re-run cluster assignment with corrected profile data
    RECOMPUTE         : Recompute benchmark with corrected input data
    DISMISS           : Invalid dispute — reason logged
  All decisions immutable and logged with full justification.

RULE_B011 — PUBLIC RANKING CONSENT RULE
  No institute may be listed in any public-facing national ranking,
  domain ranking, or regional ranking without:
    - Explicit opt-in by Institute Admin (human action)
    - Opt-in logged with actor_id + timestamp in audit trail
    - Institute achieving minimum BENCHMARK_STANDARD tier (≥ p50)
      on National Standard Benchmark Plane 1
  Institutes below p50 CANNOT opt into public rankings.
  Consent may be revoked by Institute Admin at any time.
  Revocation takes effect within 24 hours on all public surfaces.

RULE_B012 — BENCHMARK STANDARD OUTLIER PROTECTION RULE
  Outlier institutes (> 3 standard deviations from cluster mean)
  are detected and WINSORIZED (capped at 3σ boundary) before
  benchmark standard computation.
  Outlier status does NOT disqualify institute from comparison.
  Outlier flag is reported to the institute as informational.
  Winsorization applied ONLY to standard computation — individual
  institute scores are NEVER modified or capped.

RULE_B013 — CROSS-TENANT BENCHMARK DATA ISOLATION RULE
  All benchmark computations are performed on anonymized,
  aggregated data pools. No individual institute score or
  identifier is transmitted to any other institute's benchmark
  computation, API response, or dashboard surface.
  Violation → STOP EXECUTION · SECURITY INCIDENT · IMMEDIATE ESCALATION

RULE_B014 — PRELIMINARY STANDARD DISCLOSURE RULE
  If benchmark_standard displays PRELIMINARY status (RULE_B002),
  ALL dashboard surfaces, API responses, and exports MUST clearly
  label the standard as PRELIMINARY with sample_size disclosed.
  PRELIMINARY standards may NOT be used in public rankings.
  PRELIMINARY standards may NOT trigger BENCHMARK_LAGGING penalties.

RULE_B015 — BENCHMARK COMPUTATION AUDIT TRAIL RULE
  Every benchmark standard computation must produce an immutable
  audit record containing:
    computation_id
    benchmark_plane
    segment (domain / region / cluster)
    sample_size_institutes
    computation_method_version
    output_percentiles (p10 through p95)
    distribution_shape
    outlier_count_winsorized
    standard_validity_score
    computed_at
    approved_for_publication (BOOLEAN)
    approved_by (if publication required approval)
  Absence of audit record → standard NOT published.
