### A.1 — Purpose Declaration

The Antigravity Interview Semantic Analyzer (AISA) is a sealed, deterministic, multi-modal NLP engine that performs deep semantic analysis of live and recorded interview sessions within the ECOSKILLER + Dojo SaaS platform. It operates across live match rooms (WebRTC), replay sessions (signed URLs), and enterprise-integrated interview tools (HireVue, HackerRank, Codility, Zoom, Google Meet).

AISA does NOT replace human judgment. It generates structured semantic intelligence reports that feed the Scoring Engine, inform Mentor Control decisions, supply evidence for the Promotion Probability Model (APPM), and expose verified skill signals to the Talent Marketplace Trust Layer.

The **Antigravity** principle applies here as a counterforce against:

- Interviewer halo effects and unconscious bias
- Surface-level fluency masking content poverty
- Keyword stuffing masquerading as genuine competence
- Emotional performance overriding substantive answers
- Cultural and linguistic disadvantage penalizing capable candidates

AISA enforces semantic fairness — scoring **what was actually said and meant**, not how it was said or who said it.

## 🔒 SECTION B — ANTIGRAVITY SEMANTIC MECHANICS

### B.1 — What "Semantic" Means in AISA Context

Traditional interview evaluation fails at the semantic layer because it scores:

- Delivery confidence (penalizes introversion)
- Vocabulary richness (penalizes ESL candidates)
- Speed and fluency (penalizes careful thinkers)
- Interviewer familiarity bias (rewards similar-to-me patterns)

AISA decomposes every utterance into **three independent analysis layers**:

```
Layer 1 — SURFACE LAYER
  What was said (lexical content, vocabulary, fluency markers)

Layer 2 — SEMANTIC LAYER
  What was meant (intent, concept coverage, knowledge depth, reasoning structure)

Layer 3 — PRAGMATIC LAYER
  How meaning functioned in context (relevance to scenario, response appropriateness,
  argumentation quality, problem-solving approach)
```

The **Antigravity Score** is computed primarily on Layers 2 and 3, with Layer 1 used only for flagging and normalization — never for direct scoring.

This is the core antigravity mechanism: **decompress the signal from the noise**.

### C.3 — NLP Pipeline — Eight Semantic Analyzers (Locked)

Each analyzer runs independently and in parallel. No analyzer may block another.

---

#### ANALYZER 1 — Concept Coverage Analyzer (CCA)

**What it measures:** Does the candidate's response cover the key concepts required by the scenario rubric?

**Method:**

```
1. Extract rubric concept graph from Scenario Engine (per T1 Skill Validity Framework)
2. Convert candidate utterances → semantic embedding vectors (sentence-transformers: all-mpnet-base-v2)
3. Compute cosine similarity between candidate embeddings and rubric concept nodes
4. Coverage Score = (concepts_covered / total_required_concepts) × WeightedDepthBonus
```

**Output:**

```json
{
  "analyzer": "CCA",
  "concepts_required": 12,
  "concepts_covered": 9,
  "coverage_score": 0.78,
  "missing_concepts": ["concept_A", "concept_C"],
  "depth_bonus_applied": 0.05
}
```

**Rubric concept graph is version-locked per T8 Content Governance Lock.** Any rubric change triggers CCA re-calibration.

---

#### ANALYZER 2 — Reasoning Structure Analyzer (RSA)

**What it measures:** Does the candidate demonstrate structured logical reasoning, not just keyword mention?

**Method:**

```
Reasoning Pattern Detection:
  - Claim → Evidence → Warrant (Toulmin Model)
  - Problem → Analysis → Solution → Evaluation
  - Situation → Task → Action → Result (STAR pattern)
  - Hypothesis → Test → Conclusion

Scoring:
  Full structured pattern detected:     1.0 per instance
  Partial pattern (2/3 steps present):  0.6 per instance
  Unstructured assertion only:          0.2 per instance
  Contradiction detected:              -0.3 penalty

RSA_Score = mean(instance_scores) × complexity_multiplier
```

**Output:**

```json
{
  "analyzer": "RSA",
  "patterns_detected": ["STAR x2", "Claim-Evidence x3"],
  "contradictions": 0,
  "rsa_score": 0.82,
  "complexity_multiplier": 1.1
}
```

---

#### ANALYZER 3 — Knowledge Depth Analyzer (KDA)

**What it measures:** Is the candidate operating at surface recall or deep applied knowledge?

**Method:**

```
Bloom's Taxonomy Classification per utterance segment:
  Level 1 — Remember (recall facts):              weight 0.1
  Level 2 — Understand (explain concepts):         weight 0.2
  Level 3 — Apply (use knowledge in context):      weight 0.4
  Level 4 — Analyze (break down complexity):       weight 0.6
  Level 5 — Evaluate (critique, compare, judge):   weight 0.8
  Level 6 — Create (synthesize novel solutions):   weight 1.0

KDA_Score = Σ(utterance_level_weight × utterance_length_normalized) / total_segments
```

**Output:**

```json
{
  "analyzer": "KDA",
  "bloom_distribution": {
    "L1": 0.05, "L2": 0.15, "L3": 0.40,
    "L4": 0.25, "L5": 0.10, "L6": 0.05
  },
  "kda_score": 0.71,
  "dominant_level": "L3-Apply"
}
```

---

#### ANALYZER 4 — Semantic Coherence Analyzer (SCA)

**What it measures:** Does the candidate maintain consistent, coherent meaning across their full response — or do they drift, contradict, or fragment?

**Method:**

```
1. Segment candidate response into thematic blocks
2. Compute semantic similarity between consecutive blocks (sentence-transformers)
3. Detect topic drift (cosine similarity < 0.4 between adjacent blocks)
4. Detect self-contradiction (semantic contradiction detection model)
5. Compute global coherence score

SCA_Score = (1 - drift_rate) × (1 - contradiction_rate) × flow_bonus
```

**Output:**

```json
{
  "analyzer": "SCA",
  "thematic_blocks": 6,
  "drift_events": 1,
  "contradiction_events": 0,
  "drift_rate": 0.16,
  "contradiction_rate": 0.0,
  "flow_bonus": 1.05,
  "sca_score": 0.88
}
```

---

#### ANALYZER 5 — Relevance & Scenario-Fit Analyzer (RSFA)

**What it measures:** Is the candidate's response actually relevant to the specific scenario, or are they answering a generic version of the question?

**Method:**

```
1. Load scenario_context vector from Scenario Engine
2. Load candidate_response vector (full utterance embedding)
3. Compute scenario-response relevance score (cosine similarity)
4. Detect "generic answer" pattern (high fluency, low scenario-specific signal)
5. Apply specificity bonus for scenario-unique terminology usage

RSFA_Score = scenario_relevance × (1 - generic_penalty) × specificity_bonus
```

**Antigravity note:** This analyzer specifically counteracts "polished generic answerers" — candidates who sound confident but never actually engage with the specific scenario.

**Output:**

```json
{
  "analyzer": "RSFA",
  "scenario_relevance": 0.74,
  "generic_penalty": 0.08,
  "specificity_bonus": 1.06,
  "rsfa_score": 0.76
}
```

#### ANALYZER 6 — Communication Effectiveness Analyzer (CEA)

**What it measures:** How effectively did the candidate communicate their ideas — clarity, concision, signal-to-noise ratio — **NOT accent, voice, or fluency speed**.

**Locked exclusions (bias protection):**

```
EXCLUDED from CEA scoring (never scored, never logged as negative):
  - Accent or pronunciation deviation
  - Speech rate (fast vs slow)
  - Filler word count (uh, um, like)
  - Pause duration
  - Voice pitch or tone
  - Language of origin patterns
```

**Included in CEA scoring:**

```
INCLUDED (content-based communication signals):
  + Clarity: Does the idea transfer successfully? (measured via follow-up confusion signals)
  + Concision: Signal-to-word ratio (semantic density)
  + Structure: Does spoken organization match logical organization?
  + Precision: Are technical terms used correctly?
  + Adaptability: Does the candidate adjust explanation when probed?

CEA_Score = (clarity_score × 0.30) + (concision_score × 0.25) +
            (structure_score × 0.20) + (precision_score × 0.15) +
            (adaptability_score × 0.10)
```

**Output:**

```json
{
  "analyzer": "CEA",
  "clarity_score": 0.82,
  "concision_score": 0.70,
  "structure_score": 0.75,
  "precision_score": 0.88,
  "adaptability_score": 0.65,
  "cea_score": 0.77,
  "bias_exclusions_applied": ["accent", "speech_rate", "filler_words"]
}
```

#### ANALYZER 7 — Argumentation Quality Analyzer (AQA)

**What it measures:** When the candidate takes a position, do they defend it with valid arguments, or do they assert without support?

**Method:**

```
Argumentation Detection:
  1. Detect position claims ("I believe...", "The best approach is...", "We should...")
  2. For each claim, detect:
     - Supporting evidence provided?       +1.0
     - Counter-argument acknowledged?      +0.5
     - Counter-argument refuted?           +0.5
     - Claim abandoned under pressure?    -0.4
     - Claim defended with new evidence?  +0.6
     - Fallacy detected (ad hominem,
       false dichotomy, etc.)?            -0.3 per instance

AQA_Score = mean(per_claim_score) × position_count_bonus
```

**Output:**

```json
{
  "analyzer": "AQA",
  "claims_detected": 4,
  "claims_supported": 3,
  "counter_arguments_acknowledged": 2,
  "fallacies_detected": 0,
  "position_defended_under_pressure": 2,
  "aqa_score": 0.81
}
```

**What it measures:** Textual signals of emotional intelligence — empathy, self-awareness, perspective-taking — as evidenced in response content. **NOT vocal tone or facial expression.**

**Locked exclusions (bias protection):**

```
EXCLUDED from EISA scoring:

  - Vocal emotion detection (NOT performed by AISA)
```

```
  + Collaborative language ("We could approach this together...")

EISA_Score = (perspective_score × 0.30) + (stakes_score × 0.25) +

```

## 🔒 SECTION D — SEMANTIC INTELLIGENCE REPORT (SIR)

### D.1 — Master Score Computation (Immutable Formula)

```
AISA_Score = Σ(Analyzer_Score_k × Dimension_Weight_k) × ScenarioComplexityNorm × BiasAuditMultiplier
```

**Dimension Weight Registry (Locked):**

| Analyzer | Code | Default Weight | Role-Adjustable |
|----------|------|---------------|----------------|
| Concept Coverage Analyzer | CCA | 0.20 | No |
| Reasoning Structure Analyzer | RSA | 0.18 | No |
| Knowledge Depth Analyzer | KDA | 0.18 | No |

| Relevance & Scenario-Fit Analyzer | RSFA | 0.15 | No |
| Communication Effectiveness Analyzer | CEA | 0.08 | No |
| Argumentation Quality Analyzer | AQA | 0.06 | Yes (±0.02) |
| Emotional Intelligence Signal Analyzer | EISA | 0.03 | Yes (±0.02) |

**Total weights must sum to 1.0. Rebalancing requires version bump + Governance Board approval.**

---

### D.2 — ScenarioComplexityNorm

Derived from the Scenario Engine's live difficulty data (per Dojo T4):

```
ScenarioComplexityNorm = (Global_Scenario_Pass_Rate)^(-0.25)
```

- High difficulty scenario (low pass rate) → norm > 1.0 → score boosted
- Low difficulty scenario (high pass rate) → norm < 1.0 → score adjusted down
- Clamped: [0.80, 1.25]

---

### D.3 — BiasAuditMultiplier

The Bias Detection Engine (Section E) computes a multiplier that adjusts the final AISA_Score when bias risk is detected in the session:

| Bias Risk Level | Condition | Multiplier Effect |
|----------------|-----------|------------------|
| Clean | No bias signals detected | 1.00 (neutral) |
| Caution | 1–2 soft bias signals | Score flagged + 0.95 applied as caution reserve |
| Alert | 3+ signals or 1 hard signal | Score quarantined; manual review before release |
| Critical | Systematic bias pattern confirmed | Score nullified; Governance Board escalation |

## 🔒 SECTION E — BIAS DETECTION ENGINE

### E.1 — Purpose

The Bias Detection Engine runs in parallel with all 8 AISA analyzers and monitors the scoring process itself for systemic or session-level bias patterns. This is the primary trust infrastructure enforcement layer within AISA.

### E.2 — Bias Signal Registry (Locked)

| Signal ID | Bias Type | Detection Method | Severity |
|-----------|-----------|-----------------|---------|
| BS01 | Halo Effect | Correlation between early utterance scores and later dimension scores exceeds r=0.85 | Soft |
| BS02 | Linguistic Penalty | CEA score consistently lower for detected non-native speaker patterns | Hard |
| BS03 | Cultural Concept Bias | CCA missing concepts correlate with rubric concepts known to be culture-specific (T12) | Hard |
| BS04 | Gender Language Pattern | EISA scoring differs systematically based on gendered language pattern markers | Hard |
| BS05 | Session Length Bias | Longer responses scoring disproportionately higher than denser shorter responses | Soft |
| BS06 | Scenario Familiarity Advantage | RSFA specificity bonus disproportionate for industry-insider vocabulary | Soft |
| BS07 | Mentor Probe Asymmetry | Mentor follow-up probes distributed asymmetrically across candidates in same session | Hard |
| BS08 | First-Speaker Advantage | First speaker in multi-candidate sessions systematically scores higher on SCA | Soft |
| BS09 | Regional Accent Scoring Leak | ASR error rate correlates with regional origin → downstream CCA miss | Hard |
| BS10 | Complexity Over-Reward | KDA over-weighting technical jargon as depth proxy | Soft |

**Hard bias signal = immediate score quarantine.**  
**Soft bias signal = flag + caution multiplier, scoring continues with notation.**

---

### E.3 — Bias Remediation Protocol

| Severity | Immediate Action | Post-Session Action |
|----------|-----------------|---------------------|
| Soft (1–2 signals) | Flag in SIR | Bias report generated; curriculum team notified |
| Soft (3+ signals) | Caution multiplier applied | Governance Board notified; scenario fairness review triggered (T8) |
| Hard (1 signal) | Score quarantined | Manual review by independent evaluator; candidate notified of delay |
| Hard (2+ signals) | Score nullified | Governance Board escalation; session flagged for systemic audit |
| Pattern confirmed | Session series audit | Rubric revision required; all affected promotions paused |

- No skill scoring rule may be culture-exclusive

**Violation of cultural fairness lock → STOP EXECUTION of that session's scoring.**

## 🔒 SECTION F — PARALINGUISTIC EXTRACTOR (GOVERNANCE CONSTRAINED)

The Paralinguistic Extractor processes video frame and audio envelope data from WebRTC. It operates under strict governance constraints.

### F.1 — What the Paralinguistic Extractor MAY DO

```
PERMITTED:
  + Detect candidate engagement signal (speaking vs silent vs typing)
  + Detect turn-taking patterns (who speaks when)
  + Detect response latency (time from question end to answer start)
  + Detect multi-party disruption events (interruptions, crosstalk)
  + Log recording consent compliance (T10 Behavioral Safety)
```

### F.2 — What the Paralinguistic Extractor MAY NOT DO

```
FORBIDDEN (immutable, cannot be unlocked by tenant or admin):
  ✗ Facial expression scoring
  ✗ Emotion detection from video
  ✗ Eye contact measurement
  ✗ Gesture or body language analysis
  ✗ Physical appearance assessment
  ✗ Voice tone / pitch scoring
  ✗ Nervous behavior detection
  ✗ Any biometric-derived signal that flows into AISA_Score
```

**These prohibitions exist because:**

1. Such signals encode protected characteristics (race, disability, neurodivergence, culture)
2. They are not valid proxies for job-relevant skills (per T1 Construct Definition)
3. Their inclusion would violate the fundamental Antigravity principle

**Any attempt to extend Paralinguistic Extractor scope requires Governance Board supermajority vote and a new sealed version.**

### F.3 — Response Latency as a Signal

Response latency (time from question end to answer start) IS permitted but under strict governance:

```
Latency Signal Usage Rules:
  - Latency collected and stored in audit record
  - Latency NOT directly scored in AISA
  - Latency available to Mentor Panel as contextual information ONLY
  - Mentor may NOT cite latency alone as a negative in their evaluation rationale
  - Extreme latency (> 120 seconds with no response) flagged as possible technical issue
    before behavioral interpretation
```

## 🔒 SECTION G — REAL-TIME vs POST-SESSION ANALYSIS MODES

### G.1 — Real-Time Mode (Live Match Room)

During live Dojo match sessions:

```
Real-Time Analysis Flow:
  ASR (streaming) → utterance buffer → lightweight NLP pass (CCA + RSFA only)
  
Output:
  Live concept coverage indicator → Mentor Panel (read-only overlay)
  Live relevance signal → Mentor Panel (read-only overlay)
  
Scoring Engine: NOT updated in real-time (prevents live score manipulation)
Full AISA analysis: Triggered on match end event
```

**Real-time signals are advisory to Mentor Panel only. They cannot override the post-session full analysis.**

### G.2 — Post-Session Full Analysis Mode

Triggered by: `match_end` event → Recording finalize → Analytics job trigger (per Dojo P6 Integration Glue)

```
Full AISA Pipeline:
  1. Load finalized transcript from Replay Engine
  2. Run all 8 analyzers in parallel
  3. Run Bias Detection Engine
  4. Run Paralinguistic Extractor (permitted signals only)
  5. Compute AISA_Score
  6. Generate full SIR
  7. Feed APPM Component C01
  8. Notify Mentor Panel for review
  9. Write immutable audit record
  
Processing SLA: Full analysis complete within 10 minutes of session end
```

### G.3 — Async Import Mode (Enterprise Integration)

For sessions imported via EIE (HireVue, Zoom recordings, Google Meet exports):

```
Import Pipeline:
  1. EIE ingests recording + metadata
  2. Media Security: signed recording URL, time-limited access
  3. Transcription via ASR module
  4. Full AISA pipeline (same as post-session mode)
  5. SIR generated with import_source metadata
  6. Talent Marketplace employer view: SIR exposed via Skill Proof Report API
  
Note: Import-sourced SIRs are tagged with data_provenance = "external_import"
and trust_coefficient may be adjusted per Section I.2
```

## 🔒 SECTION H — AISA + DOJO SCORING ENGINE INTEGRATION

### H.1 — Score Feed Architecture

AISA feeds the Scoring Engine per the locked scoring governance (Dojo Section F):

```
AISA_Score → Scoring Engine Input:
  
  Source:         AISA (semantic dimension)
  Weight in Match Score: Configurable per rubric (default: 40% of total match score)
  Merge Logic:    Weighted merge with Peer Score + Mentor Score (Dojo Section F)
  Override Rules: Mentor override requires audit log entry (immutable)
  Variance Check: If AISA_Score deviates > 25 points from Mentor Score → anomaly flagged
  
Score Confidence:
  AISA confidence score included in SIR
  Low confidence AISA scores (< 0.70) require Mentor override for belt promotion
  (per Dojo T2 Scoring Validity Lock)
```

### H.2 — Mentor Panel Evidence Feed

The Mentor Panel receives a structured evidence summary from AISA (not raw scores):

```
Mentor Evidence Display (Flutter App — Mentor Panel):
  ├── Top 3 concepts addressed (with utterance timestamps)
  ├── Top 3 concepts missed

Mentor CANNOT see raw dimension scores before submitting their own evaluation

```

## 🔒 SECTION I — TRUST INFRASTRUCTURE ENFORCEMENT

### I.1 — AISA Confidence Score Computation

```
AISA_Confidence = (ASR_Confidence_Mean × 0.30) +
                  (Analyzer_Coverage × 0.30) +
                  (Bias_Clean_Score × 0.25) +
                  (Rubric_Match_Score × 0.15)
```

| Confidence Level | Threshold | Gate Action |
|-----------------|-----------|-------------|
| High | ≥ 0.85 | Normal pipeline; score released to APPM |
| Medium | 0.70 – 0.84 | Score released with confidence flag; Mentor review noted |
| Low | 0.55 – 0.69 | Score quarantined; Mentor Board review required before APPM feed |
| Very Low | < 0.55 | Score not released; session flagged for re-assessment |

### I.2 — Trust Coefficient for AISA → APPM

When AISA feeds APPM Component C01:

```
Trust_Coefficient_C01 = AISA_Confidence × DataProvenanceFactor × BiasCleanFactor

DataProvenanceFactor:
  Live Dojo session (primary):         1.00
  Post-session replay analysis:        0.95
  External import (EIE):               0.85
  Self-recorded / unverified:          0.70

BiasCleanFactor:
  No bias flags:                       1.00
  Soft flags only (resolved):          0.95
  Hard flags (manual review cleared):  0.85
  Active unresolved flags:             0.00 (score not fed to APPM)
```

### I.3 — Inter-Rater Reliability for AISA

AISA is treated as a "rater" in the scoring system and subject to IRR monitoring:

- AISA outputs calibrated quarterly against gold-standard scored sessions
- Drift detection: if AISA scores deviate > 8 points from gold standard mean → model re-calibration triggered
- Calibration sessions are tagged as `aisa_calibration = true` and excluded from candidate promotion counts
- AISA model version is stamped on every SIR for full reproducibility

### I.4 — Collusion & Manipulation Detection Integration (T9)

AISA contributes signals to the Manipulation Detection Engine:

| Signal | Manipulation Pattern Detected | Action |
|--------|------------------------------|--------|
| High semantic similarity between two candidates' answers in same session | Answer pre-sharing / scripting | Match flagged for audit |
| AISA_Score perfectly aligned with peer scores across multiple sessions | Score coordination | Collusion review triggered |
| Candidate RSFA score spikes only in sessions with specific mentor | Mentor-candidate favoritism | Mentor bias report generated |
| Sudden AISA_Score jump (>20 points) without corresponding rubric change | Score manipulation attempt | Governance escalation |
