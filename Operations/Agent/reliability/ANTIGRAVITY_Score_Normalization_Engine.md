## SECTION ASNE-0 — SCORE NORMALIZATION ENGINE IDENTITY DECLARATION

**Engine Name:** AntiGravity Score Normalization Engine (ASNE)
**Engine Code:** ASNE
**Engine Class:** Score Justice + Calibration + Difficulty Adjustment + Cross-Domain Fairness
**Engine Version:** v1.0
**Parent Engine:** AntiGravity Competition Forecast Engine (ACFE) v1.0
**Sibling Module:** AntiGravity Champion Authenticity Model (ACAM) v1.0
**Belongs To:** ECOSKILLER Unified Talent Operating System
**Sub-System:** Dojo SaaS Championship Module (Module 4) + Scoring Governance Layer
**AI Authority:** Computational normalization only — AI never overrides raw human-assigned scores; produces normalized scores as parallel record

**Purpose Declaration:**

Raw scores are never fair scores. A score of 85 earned in a pressure scenario under a strict mentor on a high-difficulty rubric is not the same as an 85 earned in an introductory match with a lenient evaluator. The Score Normalization Engine exists to solve this fundamental injustice. ASNE applies six normalization passes to every raw score: **Difficulty Adjustment**, **Rater Calibration Correction**, **Peer Score Variance Normalization**, **Domain Comparability Alignment**, **Pressure Context Weighting**, and **Confidence Scoring**. The output is a Normalized Score — a mathematically defensible, cross-comparable representation of true performance. Raw scores are preserved always. Normalized scores are computed in parallel. Belt promotions, championship rankings, and parent predictive AI all operate on Normalized Scores. No raw score is ever deleted. No normalization is ever hidden.

**Non-Negotiable Principle:**
Every competitor must be measured on the same mathematical playing field. The score must reflect the performance — not the evaluator, not the difficulty miscalibration, not the domain advantage. ASNE enforces that principle at every match, every round, every championship.

### PASS 1 — DIFFICULTY NORMALIZATION

**Purpose:** Adjust raw score to account for scenario difficulty so that a high score on a hard scenario reflects equally with a high score on an easy one.

```
INPUTS:
  raw_score                   (from RawScoreRecord)
  scenario_id                 (lookup → ScenarioDifficultyRecord)
  calibrated_difficulty (D)   (from ScenarioDifficultyRecord.calibrated_difficulty)
                               range: 0.50 (very easy) to 1.50 (extreme)
  global_avg_difficulty       = 1.00 (baseline — all difficulties anchor to this)

FORMULA (v1.0 — IMMUTABLE):
  difficulty_factor (DF)   = calibrated_difficulty / global_avg_difficulty
  difficulty_adjusted_score = raw_score × DF

  BOUNDS ENFORCEMENT:
    IF difficulty_adjusted_score > 100.00 → cap at 100.00
    IF difficulty_adjusted_score < 0.00  → floor at 0.00

EDGE CASES:
  IF ScenarioDifficultyRecord.sample_count < 30:
    → Use declared_difficulty_factor instead of calibrated
    → Set confidence_flags += ["SCENARIO_UNDERSAMPLED"]
    → difficulty_factor applies with 0.5 weight:
      difficulty_adjusted_score = raw_score × (1 + 0.5 × (DF - 1))

AUDIT LOG:
  pass_name = "DIFFICULTY"
  input_value = raw_score
  output_value = difficulty_adjusted_score
  adjustment_delta = difficulty_adjusted_score - raw_score
  adjustment_reason = "Scenario difficulty factor: {calibrated_difficulty}"
```

### PASS 2 — RATER CALIBRATION CORRECTION

**Purpose:** Remove systematic rater bias (leniency or harshness) from scores so that a mentor-scored match is not arbitrarily higher or lower than a peer-scored match.

```
INPUTS:
  difficulty_adjusted_score     (from Pass 1)
  rater_id + rater_type         (from RawScoreRecord)
  rater_bias_delta (BD)         (from RaterCalibrationRecord.bias_delta)
                                 = rater_avg_score - global_avg_score
                                 positive = lenient rater, negative = strict rater

FORMULA (v1.0 — IMMUTABLE):
  calibration_adjusted_score = difficulty_adjusted_score - BD

  BOUNDS ENFORCEMENT:
    IF calibration_adjusted_score > 100.00 → cap at 100.00
    IF calibration_adjusted_score < 0.00  → floor at 0.00

SPECIAL CASES:

  NEW RATER (< 10 scored matches):
    BD = 0.00 (no correction — insufficient data)
    Set confidence_flags += ["RATER_UNCALIBRATED"]
    mentor_review_required = TRUE

  UNCERTIFIED RATER:
    Score excluded from normalized calculation
    Raw score preserved but flagged
    Set confidence_flags += ["RATER_UNCERTIFIED"]
    Belt eligibility = BLOCKED until certified rater score obtained

  RATER WITH TOLERANCE_BREACH = TRUE:
    BD still applied but at 50% weight:
      calibration_adjusted_score = difficulty_adjusted_score - (BD × 0.50)
    Set confidence_flags += ["RATER_TOLERANCE_BREACH"]
    mentor_review_required = TRUE

  MENTOR OVERRIDE SCORES:
    BD applied to mentor score the same as any rater
    Mentor override authority preserved — but bias still corrected
    Override audit log entry required (from ScoringEngine)

AUDIT LOG:
  pass_name = "RATER_CALIBRATION"
  input_value = difficulty_adjusted_score
  output_value = calibration_adjusted_score
  adjustment_delta = -BD
  adjustment_reason = "Rater bias correction: rater_id={rater_id}, BD={BD}"
```

### PASS 3 — PEER SCORE VARIANCE NORMALIZATION

**Purpose:** When multiple peer evaluators score the same performance, remove statistical outliers and produce a variance-weighted peer consensus score that is robust to single extreme scores.

```
INPUTS:
  peer_scores[]               (array of {peer_id, score} from RawScoreRecord)
  mentor_score                (if present)
  calibration_adjusted_score  (from Pass 2 — for the focal scorer)
  peer_weight                 = 0.40 (default — configurable by match type)
  mentor_weight               = 0.50 (default)
  self_weight                 = 0.10 (default)

FORMULA (v1.0 — IMMUTABLE):

  Step 1 — Outlier Removal (Modified Z-Score):
    For each peer_score p_i:
      median_score = median(peer_scores[])
      MAD = median(|p_i - median_score|) for all i
      modified_z = 0.6745 × (p_i - median_score) / MAD
      IF |modified_z| > 3.5 → p_i is outlier → excluded from calculation
      Log excluded scores in confidence_flags

  Step 2 — Variance-Weighted Average:
    remaining_peers = peer_scores after outlier removal
    IF len(remaining_peers) == 0:
      → Use mentor_score only (if present)
      → Set confidence_flags += ["ALL_PEERS_OUTLIER_EXCLUDED"]

    peer_std = std_deviation(remaining_peers)
    IF peer_std == 0:
      peer_avg = mean(remaining_peers)
      variance_confidence = 1.0
    ELSE:
      variance_confidence = 1 / (1 + peer_std)  -- low variance = high confidence
      peer_avg = mean(remaining_peers)

  Step 3 — Weighted Merge:
    components = []
    IF peer_avg available:     components += peer_avg × peer_weight
    IF mentor_score available: components += mentor_score_calibrated × mentor_weight
    IF self_score available:   components += self_score × self_weight

    total_weight = sum of weights for available components
    peer_variance_score = sum(components) / total_weight  -- normalized to available

  Step 4 — Apply to pipeline:
    peer_adjusted_score = peer_variance_score

MINIMUM PEER COUNT RULES:
  < 2 peers available:
    → peer_weight redistributed to mentor_weight or flagged
    → Set confidence_flags += ["INSUFFICIENT_PEER_COUNT"]
  0 peers + 0 mentor:
    → Self-score only: not eligible for belt promotion
    → Set confidence_flags += ["SELF_SCORE_ONLY"]
    → belt_gate_status = BLOCKED_INSUFFICIENT_DATA

AUDIT LOG:
  pass_name = "PEER_VARIANCE"
  input_value = calibration_adjusted_score
  output_value = peer_adjusted_score
  adjustment_delta = peer_adjusted_score - calibration_adjusted_score
  adjustment_reason = "Peer consensus: n={peer_count}, outliers={outlier_count},
                       variance_confidence={variance_confidence}"
```

### PASS 4 — DOMAIN COMPARABILITY ALIGNMENT

**Purpose:** Enable fair cross-domain comparison in multi-domain championships. A 78 in Arts must represent the same relative performance level as a 78 in Technology, adjusted for each domain's scoring distribution.

```
INPUTS:
  peer_adjusted_score           (from Pass 3)
  domain                        (from RawScoreRecord)
  reference_domain              = "TECHNOLOGY" (universal reference — locked v1.0)
  DomainAlignmentTable          (lookup by domain pair)

FORMULA (v1.0 — IMMUTABLE):

  IF domain == reference_domain:
    domain_alignment_factor = 1.00
    domain_aligned_score = peer_adjusted_score

  ELSE:
    alignment_record = DomainAlignmentTable.lookup(domain, reference_domain)
    IF alignment_record.governance_approved = FALSE:
      → Use factor = 1.00 (no unapproved alignment)
      → Set confidence_flags += ["DOMAIN_ALIGNMENT_PENDING_APPROVAL"]
    ELSE:
      domain_alignment_factor = alignment_record.alignment_factor_a_to_b
      domain_aligned_score = peer_adjusted_score × domain_alignment_factor

  BOUNDS ENFORCEMENT:
    IF domain_aligned_score > 100.00 → cap at 100.00
    IF domain_aligned_score < 0.00  → floor at 0.00

DOMAIN ALIGNMENT TABLE UPDATE POLICY:
  Updated quarterly using pooled performance data across domains
  Minimum 500 cross-domain match samples per domain pair before factor applies
  All updates require governance board approval before activation
  Approved factors versioned alongside formula_version

SINGLE-DOMAIN CHAMPIONSHIPS:
  domain_alignment_factor = 1.00 always
  Domain alignment only meaningful for multi-domain events

AUDIT LOG:
  pass_name = "DOMAIN_ALIGNMENT"
  input_value = peer_adjusted_score
  output_value = domain_aligned_score
  adjustment_delta = domain_aligned_score - peer_adjusted_score
  adjustment_reason = "Domain alignment: {domain} → {reference_domain},
                       factor={domain_alignment_factor}"
```

### PASS 5 — PRESSURE CONTEXT WEIGHTING

**Purpose:** Scores earned under declared high-pressure scenarios (championship finals, timed sprints, adversarial opponent conditions) carry more signal about true capability than scores under no-pressure conditions. This pass weights the score to reflect that signal.

```
INPUTS:
  domain_aligned_score          (from Pass 4)
  pressure_level                (from RawScoreRecord):
                                  NONE=0, LOW=1, MEDIUM=2, HIGH=3, EXTREME=4
  match_context                 (tournament_round, belt_target)

PRESSURE WEIGHT TABLE (v1.0 — IMMUTABLE):
  NONE    → pressure_weight = 0.90  (slightly discounted — no pressure context)
  LOW     → pressure_weight = 0.95
  MEDIUM  → pressure_weight = 1.00  (neutral — baseline)
  HIGH    → pressure_weight = 1.08
  EXTREME → pressure_weight = 1.15  (championship finals, last-chance belt rounds)

FORMULA:
  pressure_weighted_score = domain_aligned_score × pressure_weight

  BOUNDS ENFORCEMENT:
    IF pressure_weighted_score > 100.00 → cap at 100.00
    IF pressure_weighted_score < 0.00  → floor at 0.00

SPECIAL CASE — CHAMPIONSHIP CONTEXT BONUS:
  IF match_context.tournament_id IS NOT NULL
  AND match_context.tournament_round IN ('SEMIFINAL', 'FINAL'):
    championship_bonus = 0.02  -- additional +2% weight
    pressure_weighted_score = pressure_weighted_score × (1 + championship_bonus)
    Re-apply bounds enforcement

PRESSURE LEVEL DECLARATION GOVERNANCE:
  Pressure level declared by MentorEngine at match setup
  Cannot be modified post-match without Governance Board approval + audit log entry
  Retroactive pressure level changes trigger re-normalization cycle

AUDIT LOG:
  pass_name = "PRESSURE_CONTEXT"
  input_value = domain_aligned_score
  output_value = pressure_weighted_score
  adjustment_delta = pressure_weighted_score - domain_aligned_score
  adjustment_reason = "Pressure weight: level={pressure_level},
                       weight={pressure_weight}"
```

### PASS 6 — CONFIDENCE SCORING

**Purpose:** Produce a confidence score (0.0–1.0) for the normalized score. Low confidence scores cannot trigger belt promotions and must be reviewed by a mentor. Confidence is derived from four factors: sample size, rater agreement, scenario calibration quality, and rater calibration health.

```
INPUTS:
  confidence_flags[]            (accumulated across Passes 1–5)
  rater_count                   (total raters who scored this match)
  variance_confidence           (from Pass 3)
  scenario_sample_count         (from ScenarioDifficultyRecord)
  rater_calibration_score       (from RaterCalibrationRecord)
  irr_cronbach_alpha            (from InterRaterReliabilityRecord for this scenario)

CONFIDENCE FACTOR COMPUTATION (v1.0 — IMMUTABLE):

  Factor 1 — Sample Size Factor (SSF):
    IF rater_count >= 4: SSF = 1.00
    IF rater_count == 3: SSF = 0.85
    IF rater_count == 2: SSF = 0.70
    IF rater_count == 1: SSF = 0.50
    IF rater_count == 0: SSF = 0.00  → STOP: insufficient data

  Factor 2 — Rater Agreement Factor (RAF):
    RAF = variance_confidence (from Pass 3 output)
    RAF range: 0.0 – 1.0

  Factor 3 — Scenario Stability Factor (SSF2):
    IF scenario_sample_count >= 100: SSF2 = 1.00
    IF scenario_sample_count >= 50:  SSF2 = 0.85
    IF scenario_sample_count >= 30:  SSF2 = 0.70
    IF scenario_sample_count < 30:   SSF2 = 0.55

  Factor 4 — Rater Calibration Factor (RCF):
    RCF = rater_calibration_score (from RaterCalibrationRecord)
    IF rater_type = PEER and no calibration record: RCF = 0.60
    IF rater_type = MENTOR and no calibration record: RCF = 0.50
    IF tolerance_breach = TRUE: RCF = RCF × 0.70

  CONFIDENCE FLAG PENALTIES:
    Each flag in confidence_flags[] applies penalty:
    "SCENARIO_UNDERSAMPLED"         → multiply confidence × 0.85
    "RATER_UNCALIBRATED"            → multiply confidence × 0.80
    "RATER_UNCERTIFIED"             → multiply confidence × 0.50
    "RATER_TOLERANCE_BREACH"        → multiply confidence × 0.75
    "INSUFFICIENT_PEER_COUNT"       → multiply confidence × 0.80
    "ALL_PEERS_OUTLIER_EXCLUDED"    → multiply confidence × 0.70
    "SELF_SCORE_ONLY"               → confidence = 0.00
    "DOMAIN_ALIGNMENT_PENDING"      → multiply confidence × 0.90

COMPOSITE CONFIDENCE FORMULA:
  confidence_score = (
    0.30 × SSF   +
    0.30 × RAF   +
    0.20 × SSF2  +
    0.20 × RCF
  ) × flag_penalty_multiplier

  CONFIDENCE INTERPRETATION:
    confidence_score ≥ 0.80  → HIGH
    0.65 ≤ score < 0.80      → MEDIUM
    0.40 ≤ score < 0.65      → LOW   → mentor_review_required = TRUE
    score < 0.40             → INSUFFICIENT → belt_gate_status = BLOCKED

BELT GATE RULE (NON-NEGOTIABLE):
  IF confidence_score < 0.65:
    → is_low_confidence = TRUE
    → belt_eligible = FALSE
    → Belt promotion BLOCKED pending mentor board review
    → Low confidence flag dispatched to mentor
    → Parent notified via ParentScoreInsight (plain language)

FINAL NORMALIZED SCORE COMPUTATION:
  normalized_score = pressure_weighted_score  (output of Pass 5)
  (Confidence does not modify the score value — it gates belt eligibility only)

PERCENTILE RANK COMPUTATION:
  Compare normalized_score against domain + belt_cohort distribution
  percentile_rank = percentile_rank_in_cohort(normalized_score, domain, belt_level)

Z-SCORE COMPUTATION:
  z_score = (normalized_score - cohort_mean) / cohort_std_deviation
  IF cohort_size < 10: z_score = NULL (insufficient cohort)

AUDIT LOG:
  pass_name = "CONFIDENCE"
  input_value = pressure_weighted_score
  output_value = normalized_score
  adjustment_delta = 0.00  (confidence does not change score value)
  adjustment_reason = "Confidence: {confidence_score}, flags: {confidence_flags[]}"
```

## SECTION ASNE-5 — RATER CALIBRATION SUBSYSTEM (LOCKED)

### Monthly Calibration Cycle

```
CYCLE TRIGGER: First day of each month (scheduled job)
SCOPE: All active raters (peers with ≥ 10 scored matches + all mentors)

STEP 1 — Gold Standard Match Set:
  System maintains a library of Gold Standard Matches:
    - Matches previously scored by certified Governance Board evaluators
    - Stored with official_score per rubric metric
    - Minimum 20 gold standard matches per domain per belt level

STEP 2 — Calibration Task:
  Each rater assigned 3–5 gold standard matches to evaluate blind
  Rater submits scores without seeing official scores

STEP 3 — Calibration Score Computation:
  calibration_score = 1 - (|rater_score - official_score| / max_possible_score)
  Mean calibration_score across all tasks = rater_calibration_score

STEP 4 — Bias Delta Computation:
  bias_delta = rater_avg_score_last_90days - global_avg_score_last_90days

STEP 5 — Drift Detection:
  drift_from_last_cycle = |current_bias_delta - previous_bias_delta|
  IF drift_from_last_cycle > 0.15 → drift_flag = TRUE

STEP 6 — Tolerance Breach:
  IF |bias_delta| > 15% of max_possible_score → tolerance_breach = TRUE
  IF tolerance_breach:
    → Alert dispatched to mentor supervisor
    → Scores from this rater in next cycle weighted at 50%
    → Recertification required within 30 days

STEP 7 — Mentor Recertification Trigger:
  IF tolerance_breach = TRUE for 2 consecutive cycles:
    → recertification_required = TRUE
    → Mentor loses certification authority on ranked matches
    → Governance Board notified

INTER-RATER RELIABILITY COMPUTATION (per scenario per cycle):
  Collect all rater scores for same scenario across past 30 days
  Compute Cronbach's Alpha:
    α = (k / (k-1)) × (1 - Σσ²_i / σ²_total)
    where k = number of raters, σ²_i = variance of rater i
  Compute Intraclass Correlation Coefficient (ICC)
  IF cronbach_alpha < 0.70 → low_irr_flag = TRUE → governance review needed

GOLD STANDARD MATCH GOVERNANCE:
  All gold standard matches versioned and immutable after official scoring
  New gold standard matches added via content ops governance workflow
  Minimum 20 new gold standard matches added per domain per belt level per year
```

### Rater Calibration Score Impact on Normalization

```
Calibration Health → Rater Calibration Factor (RCF) → Confidence Score → Belt Gate

Chain:
  Poor calibration → Low RCF → Low Confidence → Belt blocked → Mentor review required

Recovery path:
  Rater completes recalibration → New calibration score → RCF updates → 
  New normalized scores computed for affected matches → Belt eligibility re-evaluated

Retroactive Correction Policy:
  IF rater recertified after tolerance breach:
    → All matches scored during breach period re-normalized with corrected bias_delta
    → NormalizedScoreRecord updated (new record created — old preserved)
    → Audit log entry: "Retroactive normalization: rater recertified {date}"
    → Affected users notified
    → Parent AI scores refreshed
```

## SECTION ASNE-6 — SCENARIO DIFFICULTY CALIBRATION (LOCKED)

### Difficulty Calibration Algorithm

```
RECALIBRATION TRIGGER:
  Every 30 days (scheduled) OR when:
    pass_rate deviates > 10% from last calibration
    abandonment_rate > 30%
    failure_cluster_flag raised

CALIBRATION INPUTS (minimum 30 completed attempts required):
  pass_rate:              % of attempts scoring ≥ 60% of max
  avg_score:              mean score across all attempts
  score_std_deviation:    variability measure
  avg_completion_time:    vs. declared time limit
  abandonment_rate:       % starting but not finishing
  failure_cluster:        detected if ≥ 40% fail at same scenario step

CALIBRATED DIFFICULTY FORMULA (v1.0 — IMMUTABLE):
  base_difficulty = (1 - pass_rate) × 0.50 +
                    (1 - (avg_score / 100)) × 0.30 +
                    (abandonment_rate) × 0.20

  calibrated_difficulty = 0.50 + (base_difficulty × 1.00)
  -- Range: 0.50 (trivial, 100% pass) to 1.50 (extreme, 0% pass)

  BOUNDS:
    calibrated_difficulty = max(0.50, min(1.50, calibrated_difficulty))

DIFFICULTY LABEL RECLASSIFICATION:
  calibrated_difficulty < 0.70  → reclassify as BEGINNER
  0.70 ≤ value < 0.90           → reclassify as INTERMEDIATE
  0.90 ≤ value < 1.15           → reclassify as ADVANCED
  value ≥ 1.15                  → reclassify as EXPERT

  RULE: Difficulty labels must be data-derived. Author-declared labels are
        overridden by calibrated values after minimum sample count reached.

SCENARIO RETIREMENT RULES:
  IF pass_rate > 95% for 3 consecutive cycles → retirement_recommended = TRUE
    (Too easy — no discriminative value)
  IF abandonment_rate > 50% for 2 cycles → retirement_recommended = TRUE
    (Too discouraging — fairness concern)
  IF failure_cluster_flag for 3 cycles → governance_review_needed = TRUE
    (Systematic unfairness at specific step)

SCENARIO FAIRNESS AUDIT:
  Cross-domain analysis: same scenario pass_rate across domains
  IF pass_rate variance across domains > 20% → domain_bias_flag raised
  Content governance team reviews scenario for cultural/domain neutrality
```

  rater_calibration_score = 0.88  (well-calibrated mentor)

    peer component:   69.67 × 0.40 = 27.87
    mentor component: 75.00 × 0.50 = 37.50   (calibrated mentor score used)
    self: not submitted
    total_weight = 0.90
  peer_variance_score = (27.87 + 37.50) / 0.90 = 72.63
  → Output: 72.63

  DomainAlignmentTable: Commerce → Technology factor = 1.03
