## SECTION 4 — TRUTH VIOLATION TAXONOMY (SEALED)

All detected violations must be classified into exactly one of these 8 violation categories. No item may be left unclassified.

### Category 1: REALITY LAW VIOLATION (RLV)
Directly violates one of the six M-Laws or the seven L-Laws. The most serious class.

| Sub-type | Code | Description |
|---|---|---|
| M1 Execution Claim | RLV-M1 | Artifact claims cloud provisioning, Terraform apply, K8s cluster creation completed |
| M2 Business Finalization | RLV-M2 | Pricing, legal jurisdiction, or payment contracts presented as finalized/activated |
| M3 UI Polish Claim | RLV-M3 | UI artifact claims final design quality, pixel perfection, or production UX readiness |
| M4 Performance Claim | RLV-M4 | Architecture claims performance/scalability without human-executed test results |
| M5 Model Claim | RLV-M5 | AI/ML component claims to be trained, deployed, or making live predictions |
| M6 Operations Claim | RLV-M6 | Monitoring claims to be active; alerts claim to be live; ops status claimed without runtime evidence |
| L-Law Deployment Claim | RLV-L | Artifact claims "Live SaaS", "Deployed", or "Production Operational" without L3 verification |

### Category 2: FALSE PRODUCTION LABEL (FPL)
A "No system may claim…" seal is violated. The artifact self-applies a forbidden label without the backing law being verified.

| Sub-type | Code | Description |
|---|---|---|
| Performance label | FPL-PERF | Claims "Fast", "Realtime", "Scalable" without R49 verified |
| Scalability label | FPL-SCALE | Claims "Internet scale", "Enterprise growth ready" without R50 verified |
| Observability label | FPL-OBS | Claims "Observable", "Production-ready" (observability context) without R51 verified |
| Audit/Compliance label | FPL-AUDIT | Claims "Auditable", "Compliant", "Trustworthy" without R52 verified |
| Release/CI label | FPL-REL | Claims "Safe to update", "Continuously delivered" without R53 verified |
| Config label | FPL-CONF | Claims "Stable", "Well-governed" without R55 verified |
| Vendor/Security label | FPL-SEC | Claims "Enterprise-ready", "Secure", "Zero-trust aligned" without R56/R57/R58/R59/R60 verified |
| Global readiness label | FPL-GLOB | Claims "Production-Ready", "Enterprise-Ready", "Live SaaS" without full law stack verified |

### Category 3: ARTIFACT COMPLETENESS VIOLATION (ACV)
The artifact is not what it claims to be — it is pseudocode, a stub, a placeholder, an interface-only shell, or incomplete generation presented as a complete artifact.

| Sub-type | Code | Description |
|---|---|---|
| Pseudocode disguised | ACV-PS | Code block written as pseudocode/English steps instead of executable code |
| Stub/placeholder | ACV-ST | Function exists but body is `pass`, `TODO`, `NotImplemented`, `raise NotImplementedError`, or placeholder comment |
| Interface-only | ACV-IO | Class/interface declared but no implementation provided |
| Missing DB wiring | ACV-DB | Service claims to persist data but has no ORM model, migration, or repository layer |
| Non-runnable | ACV-NR | Artifact has syntax errors, broken imports, or missing dependencies that prevent execution |
| Missing auth connection | ACV-AU | Route claims authentication but no auth middleware is wired |
| Mock masquerading | ACV-MK | Hardcoded mock data returned where real persistence is required |
| Missing event publish | ACV-EV | Service claims event-driven behavior but produces no Kafka publish calls |
| Partial generation | ACV-PG | Only subset of declared entities/endpoints/screens were generated; remainder missing without explicit gap declaration |

### Category 4: CONTRACT DRIFT VIOLATION (CDV)
The artifact's declared contract (OpenAPI spec, AsyncAPI schema, permission matrix, design tokens) does not match its actual implementation.

| Sub-type | Code | Description |
|---|---|---|
| API spec drift | CDV-API | OpenAPI spec declares endpoint/schema that differs from generated route/model |
| Event schema drift | CDV-EVT | AsyncAPI schema declares event structure that differs from generated Kafka publisher payload |
| Permission matrix drift | CDV-PRM | Permission matrix declares role→screen mapping that is not enforced in generated RBAC middleware |
| Design token drift | CDV-TOK | Design tokens file declares color/spacing/font that is overridden in generated UI component |
| Domain mapping drift | CDV-DOM | `/config/domain_mapping.yaml` declares URL pattern that differs from generated Ingress/route |
| Schema registry drift | CDV-SCH | DB entity schema declared in registry differs from generated Alembic migration |
| UI component registry drift | CDV-UIC | UI Component Contract Registry declares component props/behavior that differs from generated component |

### Category 5: CONFIGURATION TRUTH VIOLATION (CTV)
Configuration declarations do not match runtime configuration, environment config has drifted, or config source-of-truth is violated.

| Sub-type | Code | Description |
|---|---|---|
| Config drift | CTV-DR | Runtime configuration diverged from declared source-of-truth config |
| Environment leak | CTV-EL | DEV config value appears in STAGING/PROD config |
| Hardcoded env override | CTV-HO | Configuration hardcoded in code instead of sourced from config system |
| Config mutation | CTV-MU | Runtime system is mutating the source-of-truth config (R55.4 violation) |
| Stale config | CTV-SC | Config file references deprecated or removed system components |
| Missing env isolation | CTV-MI | Environment-specific config not separated (same config used for multiple environments) |

### Category 6: DOCUMENTATION TRUTH VIOLATION (DTV)
Documentation, README files, comments, or labels make claims that are unsubstantiated, inconsistent with code, or aspirational rather than descriptive.

| Sub-type | Code | Description |
|---|---|---|
| Aspirational diagram | DTV-AS | Architecture diagram shows future state mixed with current state without clear labeling |
| Misleading README | DTV-RM | README claims a feature is "implemented" or "working" when it is a stub or placeholder |
| False status label | DTV-SL | Status badge, label, or inline comment claims "COMPLETE", "DONE", "PRODUCTION" without evidence |
| Contradictory docs | DTV-CD | Two documents describe the same component with contradictory behavior |
| Stale documentation | DTV-SD | Documentation describes behavior that has changed in the code |
| Undocumented behavior | DTV-UB | Code does something not described anywhere in any documentation |
| Generic certification claim | DTV-CC | Generic claim of compliance certification not backed by actual certification (e.g., "SOC2 compliant" without audit) |

### Category 7: AI SELF-CLAIM VIOLATION (ASV)
The AI generator itself made a claim in generated commentary, docstrings, README content, or inline annotations that violates the reality laws or makes unsubstantiated assertions about the system's capabilities.

| Sub-type | Code | Description |
|---|---|---|
| AI completion false claim | ASV-CF | AI-generated comment or text claims task is "complete" when inspected code shows otherwise |
| AI deployment claim | ASV-DC | AI-generated text claims system is deployed, live, or running in production |
| AI performance assertion | ASV-PA | AI-generated docstring or comment asserts specific performance metrics without test evidence |
| AI security assertion | ASV-SA | AI-generated comment asserts code is "secure", "hardened", or "OWASP-compliant" without evidence |
| AI coverage assertion | ASV-CA | AI-generated text asserts "100% coverage" or "fully tested" without attached test results |
| AI fabricated evidence | ASV-FE | AI-generated content fabricates log outputs, test results, or runtime data as examples presented as real |
| AI scope inflation | ASV-SI | AI-generated output implies broader completion than was actually requested or delivered |

### Category 8: SOURCE OF TRUTH VIOLATION (STV)
Multiple sources claim authority over the same truth, or the single source of truth is bypassed, duplicated, or inconsistent.

| Sub-type | Code | Description |
|---|---|---|
| Dual truth sources | STV-DT | Two different files both claim to be the authoritative source for the same value |
| Truth source bypassed | STV-TB | Code reads value from non-authoritative source instead of declared source of truth |
| Duplicated entity definition | STV-DE | Same entity/schema/contract defined in two places with potential drift between them |
| Inferred baseline | STV-IB | Ad-hoc or inferred baseline used instead of versioned declared state (Drift Law Section C) |
| Runtime mutation of truth | STV-RM | Runtime system modifying the source-of-truth file/registry (R55.4 violation) |

### DIMENSION TE-3: ARTIFACT COMPLETENESS VERIFICATION
**Laws:** R44, R45, R38, R46, R48, R24, R26 | **Weight:** 12%

**TE-3.1 — Non-runnable backend scan (R44 enforcement)**

- Check: every `@router.get/post/put/delete` has a real handler body (not `pass`, `TODO`, `raise NotImplementedError`)

- Check: every handler that claims auth has a `Depends(get_current_user)` or equivalent middleware call

**TE-3.2 — Stub & placeholder detection**
Scan every `.py` file for:

- `raise NotImplementedError`
- `return {}` or `return []` in any handler that is documented as returning real data
- `...` (Ellipsis) as function body outside Protocol/abstract class context
- String-only function bodies like `"""Stub implementation"""`

**TE-3.3 — Mock data masquerading (ACV-MK)**
- Any hardcoded list or dict returned where the route is documented as DB-backed
- Pattern: `return [{"id": 1, "name": "Test User"}]` in a production route

**TE-3.4 — Missing Kafka publish detection (ACV-EV)**
