Which STT Engine to Use — Full Comparison

⚡ Vosk — vosk-model-hi MVP

Streaming-first, 50MB Hindi model, runs on any CPU. Fastest real-time response (~50ms). Designed for telephony.

  • Streaming API — no chunking needed
  • Runs on cheapest VPS (1 vCPU, 1GB RAM)
  • Dedicated Hindi model available
  • Used in call centers worldwide

Downside: Less accurate on Hinglish, heavy accents, noisy PSTN audio.

🎯 Whisper.cpp — small model Quality

OpenAI's Whisper in C++. Best accuracy for Hindi + Hinglish. Runs on CPU. MIT license.

  • Handles noisy 8kHz PSTN audio well
  • Robust on UP, Bihar, MP, Rajasthan accents
  • 99 languages — handles Hinglish perfectly
  • Small model: 244MB, 2-3s latency

Downside: Not truly streaming — processes in 2-3s chunks.

🚀 Mistral Voxtral Mini (3B) Best 2025

New in 2025. Apache 2.0. Outperforms Whisper Large V3 AND GPT-4o mini Transcribe on Hindi benchmarks.

  • Built-in language understanding, not just STT
  • Can answer: "what was speaker's main argument?"
  • Apache 2.0 — completely free, commercial OK
  • 3B model runs on CPU (needs 4-8GB RAM)

Downside: Higher RAM requirements, higher server cost.

🌐 Faster-Whisper Balanced

Community-optimized Whisper via CTranslate2. 4x faster than original with same accuracy.

  • 4x faster, 2-3x lower memory than Whisper
  • Real-time friendly with chunked streaming
  • Same accuracy as Whisper large
  • Most used in production 2025

Downside: Still 2s window, not true streaming.

Security & PIN Protection NEW

⚠ MISSING FROM PREVIOUS REPORT — CRITICAL FOR PRODUCTION
🚨

A 6-digit PIN has only 1,000,000 combinations. Without protection, a bad actor can brute-force your way into any live GD session in minutes. This section covers every security layer needed before going live.

🔒 Threat 1: PIN Brute-Force Attack

Risk: Attacker writes a script that calls your number and tries 000000, 000001, 000002... until they find a valid session PIN and join someone else's GD.

Solution — 3 layers of protection:

  • Rate limit: Max 3 wrong PIN attempts per phone number per 10 minutes — then block the number for 30 minutes
  • Lockout: After 5 total wrong attempts on a session PIN, disable that PIN entirely and notify the session host
  • PIN expiry: Session PIN expires automatically 5 minutes after session ends — reduces attack window

🔒 Threat 2: Session Hijacking via Caller ID Spoofing

Risk: Attacker spoofs a legitimate participant's phone number to bypass caller-ID based filters.

Solution:

  • Never rely on Caller ID as the only auth — always require PIN
  • Use PIN + phone number whitelist (optional) for high-security sessions
  • Log all joining attempts with timestamp + Caller ID + IP of FreeSWITCH gateway

⏰ Threat 3: Stale PINs Being Reused

Risk: A PIN from a GD session 2 days ago still works and someone joins a new session that happened to reuse the same PIN.

Solution:

  • Generate PINs with a UUID-based seed, not sequential numbers
  • Hard expiry: PIN becomes invalid when session status = "ended"
  • Add session creation timestamp to PIN uniqueness check — same 6 digits cannot repeat within 24 hours

🔇 Threat 4: Spam/Abuse Calls

Risk: Someone repeatedly dials your conference number with garbage to waste your SIP trunk minutes (costs you money) or disrupt a live session.

Solution:

  • Global rate limit: max 10 calls per hour from same number to your service
  • Session max capacity: define max phone participants per session (e.g., 3)
  • Moderator can hard-ban a phone number from the app UI
  • Auto-hangup after 3 wrong PINs from same call

✅ Security Implementation Checklist

  • Rate limit: 3 wrong PINs per number per 10min
  • Session lockout after 5 total wrong attempts
  • PIN auto-expires when session ends
  • No sequential PINs — use cryptographically random
  • Log all join attempts (number, time, success/fail)
  • Max phone participants cap per session
  • Moderator can ban phone numbers in real-time
  • Auto-hangup on repeated wrong PIN from same call

Latency & Audio Sync Problem NEW

⚠ MISSING FROM PREVIOUS REPORT — REAL PRODUCTION ISSUE
⚠️

PSTN calls introduce a natural 200–400ms delay compared to WebRTC's ~50ms. This means the phone user hears app users slightly delayed, and vice versa. Without managing this, users will experience echo, double-talk, and awkward overlaps. Here's how to handle it properly.

Latency Comparison

📱

WebRTC Users

~50ms

Near-instantaneous. Best case. All app users at this latency.

☎️

PSTN Phone User

200–400ms

Caused by PSTN encoding, SIP processing, FreeSWITCH mixing. Noticeable but manageable.

🗣️

STT Processing

+50ms – 2s

Vosk: +50ms. Whisper: +2-3s chunks. Parallel to audio — doesn't block the call.

🔧 Fix 1: Echo Cancellation (Critical)

Without echo cancellation, the phone user will hear their own voice echoed back ~400ms later. This is highly disruptive.

  • Enable FreeSWITCH mod_echo_cancellation — built-in, free, one config line
  • For calls from BSNL (known echo issues), add aggressive echo suppression level 3
  • FreeSWITCH conference already does acoustic echo cancellation (AEC) by default when configured correctly

🔧 Fix 2: Jitter Buffer Management

Rural 2G calls have inconsistent packet timing (jitter). Without a buffer, audio sounds choppy and broken to all participants.

  • Configure FreeSWITCH jitter buffer: jitter-buffer-msec=60 in conference profile
  • Set adaptive buffer that increases automatically if packet loss detected
  • Adds ~60ms latency but gives smooth audio even on unstable 2G signals

🔧 Fix 3: Audio Quality Upsampling

Phone user audio comes in at 8kHz. App users are at 48kHz. When mixed, the phone user sounds "muffled" compared to everyone else.

  • FreeSWITCH automatically resamples 8kHz → 48kHz during conference mixing
  • Enable mod_samplerate with Speex resample algorithm (better quality than default)
  • Result: phone user's voice sounds cleaner to app users

🔧 Fix 4: Double-Talk Prevention

Because of the 200-400ms delay gap, phone users often accidentally interrupt because they can't hear that someone already started speaking.

  • In structured GDs: use push-to-talk for phone users — they press * to speak
  • In free-form GDs: add a 200ms "hold-off" before phone user's audio is unmuted
  • Show app users a "⚠️ Phone user may hear you delayed" notice in UI
✅

Realistic expectation: With all fixes applied, phone user experience will be similar to a speakerphone on a Group Call. Perfectly usable for GD. The 200–400ms delay is perceptible but not conversation-breaking. Most rural users are already accustomed to this delay from normal conference calls on BSNL/MTNL.

Score Fairness — 8kHz vs 48kHz Problem NEW

⚠ MISSING FROM PREVIOUS REPORT — AFFECTS CORE PRODUCT FAIRNESS
⚠️

Phone users call in at 8kHz audio (PSTN narrowband). App users stream at 48kHz (WebRTC wideband). This quality difference directly causes STT to produce lower-quality transcripts for phone users — fewer words recognized, more errors — which means the AI will unfairly score phone users lower. This is a product integrity issue.

❌ Without Fairness Correction

  • Phone user says "climate change mitigation" — STT hears "climate hang mitigation" (8kHz garble)
  • STT Word Error Rate: ~15-20% for phone users vs ~3-5% for app users
  • AI penalizes phone user for "lower quality speech" when it's actually audio quality
  • Phone user gets 58/100 score; same performance app user gets 75/100 — unfair
  • Rural students disadvantaged purely because of infrastructure

✅ With Fairness Correction Applied

  • System detects participant type: phone vs app at join time
  • Apply "phone participant" flag to all their transcription data
  • STT confidence threshold lowered for phone users (accept lower-confidence words)
  • Apply +10–15% score normalization for known audio quality disadvantage
  • Show "📞 Joined via phone call" note in their report so evaluators are aware

🔧 Technical Fixes for Score Fairness

🎚️

FreeSWITCH Preprocessing

Apply wideband simulation (bandwidth extension) before sending 8kHz audio to STT. mod_opus can upsample PSTN audio. Adds ~20ms latency but significantly improves STT accuracy.

📊

Score Normalization Algorithm

Track historical STT accuracy per connection type. If phone users consistently get 15% lower STT word confidence, apply 15% upward normalization to their linguistic quality score. Recalibrate monthly.

🏷️

Transparent Disclosure

Show in report: "Score calculated with phone-call audio adjustment applied." Both the student and evaluator know the context. This builds trust rather than hiding the normalization.

🎯

Important: Score normalization must be based on measured data, not a hardcoded offset. Measure actual STT error rates on 100+ real phone calls from Indian telecom networks, then derive the correction factor empirically. This makes the scoring defensible to users.

Data Privacy & Legal Compliance NEW

⚠ MISSING FROM PREVIOUS REPORT — LEGAL REQUIREMENT IN INDIA
⚖️

India's Digital Personal Data Protection Act 2023 (DPDP Act) came into force in 2023. Recording phone calls, storing transcripts, and retaining phone numbers without consent is a legal violation with potential fines. This must be addressed before production launch.

📞 Call Recording Consent

You MUST inform callers they are being recorded. The IVR must say: "This call may be recorded for analysis purposes. Press 1 to consent and continue. Press 2 to join without recording (analysis will be limited)." Consent must be logged with timestamp.

📝 Transcript Data Retention

Under DPDP Act, personal data cannot be retained longer than necessary. Policy: Transcripts retained for 30 days after session, then auto-deleted. Raw audio deleted within 24 hours of session end. User can request earlier deletion via SMS reply "DELETE".

📱 Phone Number Storage

Phone numbers are personal data under DPDP. You cannot store them without explicit purpose declaration. In your privacy policy: phone numbers stored only for report delivery and session management, not shared with third parties, deleted after 90 days of inactivity.

🔐 Audio Data Security

Call recordings must be encrypted at rest (AES-256). Access to call recordings restricted to authorized personnel only, with audit log. FreeSWITCH can be configured to write recordings directly to encrypted storage.

👶 Minor Protection

EcoSkiller likely serves students including minors under 18. Under DPDP, processing personal data of minors requires parental consent. Add age verification in user registration and treat all unverified phone users as potential minors — apply stricter data handling.

📋 What Your Privacy Policy Must Say

Explicitly state: (1) What data is collected from phone users, (2) How long it's retained, (3) Who has access, (4) How to request deletion, (5) That calls may be recorded and transcribed, (6) Right to withdraw consent. Display this at ecoskiller.com/privacy and link from the SMS that sends the PIN.

✅

Quick compliance checklist: IVR consent prompt ✓ | 30-day auto-delete transcripts ✓ | 24h audio delete ✓ | Encrypted storage ✓ | Privacy policy page ✓ | Deletion request via SMS ✓

Failover & Reliability — What Happens When Server Crashes NEW

⚠ INCOMPLETE IN PREVIOUS REPORT
🔥

A rural user walked away from internet connectivity, joined via phone, is mid-GD — and your FreeSWITCH server crashes. The call drops. They cannot re-join from the app. They have no idea what happened. Without a failover plan, this destroys trust with exactly the users you're trying to help.

🚨 Failure Scenarios to Plan For

  • FreeSWITCH process crashes mid-session
  • VPS goes down (host maintenance, OOM kill)
  • SIP trunk provider outage (MCUBE downtime)
  • Network connectivity lost between VPS and SIP trunk
  • STT service crashes — analysis stops but call continues
  • Database unavailable — PIN validation fails for new joiners

✅ Mitigation Strategies

  • Watchdog process: Supervisor/systemd auto-restarts FreeSWITCH within 10 seconds of crash
  • Session state in Redis: If FS crashes, Redis still has PIN→session mapping. Restart re-reads state
  • Auto-reconnect: After restart, system auto-calls back known phone participants (already in advanced features)
  • STT is non-blocking: If STT crashes, call continues — analysis is paused, not the GD
  • Backup SIP trunk: Configure 2 SIP providers. If MCUBE is down, failover to Tata/Knowlarity automatically

✅ IVR Design Best Practices

  • Always use human-recorded voice for Hindi, not TTS (TTS sounds robotic on PSTN)
  • Maximum 2 retry attempts before any action — don't frustrate users
  • State the number of remaining attempts clearly after each failure
  • Always give a recovery path — never dead-end the user
  • Confirm successful join with audio confirmation, not silence

WebRTC ↔ FreeSWITCH Bridge Integration NEW

⚠ INCOMPLETE IN PREVIOUS REPORT — THE HARDEST TECHNICAL PART
🔗

The original report explained how phone users join via FreeSWITCH, and separately how app users use WebRTC — but never explained how these two systems are connected to each other. This is actually the hardest integration step. There are 3 architecture options, each with different complexity and trade-offs.

❌ The Problem

Your EcoSkiller app users are in a WebRTC room (e.g., Daily.co, Agora, or your own WebRTC server). Phone users join a FreeSWITCH conference room. These are two completely different audio systems. Without an explicit bridge, app users and phone users are in entirely separate audio sessions and cannot hear each other.

3 Architecture Options — Choose One

Option A: Full FreeSWITCH (Simplest) Recommended

Idea: Migrate ALL participants (app users too) to FreeSWITCH conference. App users connect via WebRTC-SIP using their browser. Phone users connect via PSTN-SIP. Everyone is in the same FreeSWITCH conference room — no bridge needed.

Advantages

  • Simplest architecture — one system for everyone
  • No bridge to maintain or troubleshoot
  • FreeSWITCH natively supports WebRTC via WebSocket SIP
  • Works with standard SIP.js in browser (open source)

Disadvantages

  • You must migrate existing WebRTC infrastructure
  • Loss of features from Daily.co/Agora (video, recording, etc.)
  • FreeSWITCH WebRTC setup is complex — requires STUN/TURN config

Option B: SIP Gateway Bridge (Balanced) Good for Existing Apps

Idea: Keep your existing WebRTC (Daily.co / Agora / Jitsi) for app users. Run FreeSWITCH separately for phone users. Use a SIP-WebRTC gateway to bridge the two. Audio from FreeSWITCH conference is forwarded to WebRTC room and vice versa.

  • Dependency on Twilio vendor — not 100% open source
  • Higher cost than Options A/B at scale
