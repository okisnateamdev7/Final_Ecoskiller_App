SECTION 01 — THE CORE IDEA: WHAT THIS IS
================================================================================

This system is called a PSTN-to-WebRTC Bridge. It is the same technology used
behind Zoom and Google Meet dial-in numbers. A phone call from a 2G phone is
converted into a WebRTC audio stream and injected into a live Group Discussion
(GD) room. The phone caller hears everyone in the room, and everyone in the room
hears the phone caller — in real-time. No AI fake participants. It is the exact
same live session.

HOW A PHONE USER JOINS THE LIVE GD SESSION:
  📱 Rural User (No internet, 2G voice only)
     → Dials a Number (Any phone, no app needed)
     → Goes through PSTN / GSM (Normal call, 2G works fine)
     → FreeSWITCH (Open source, FREE bridge)
     → Enters the Live GD Room (Same room as app users)
     → App Users hear the phone user in real-time

IMPORTANT: NO FAKE AI — THE PHONE USER IS A REAL PARTICIPANT IN THE ACTUAL
LIVE SESSION.

WHAT WE ARE NOT BUILDING:
  ✗ A separate "offline GD" where user talks alone
  ✗ AI-simulated fake discussion partners
  ✗ Post-session analysis only (after call ends)
  ✗ A new app the user must install
  ✗ Something that needs internet on the user's phone

WHAT WE ARE BUILDING:
  ✓ User dials a number — instantly joins live GD
  ✓ Hears all app users in real-time
  ✓ App users hear the phone user in real-time
  ✓ AI scores the phone user's participation live
  ✓ Report delivered via SMS/WhatsApp after session

Every feature below is technically possible with the open-source stack. Core

--- CORE CAPABILITIES (Must Have) ---

1. PHONE DIAL-IN TO LIVE SESSION
   User calls a number, enters a PIN, and is instantly bridged into the exact
   live GD room. They hear and speak in real-time — just like Zoom dial-in.
   [Core Feature]

2. IVR IN HINDI & ENGLISH
   Automated voice greets the caller in Hindi or English, asks for session PIN,
   confirms joining. Fully customizable prompts.
   [Core Feature]

3. REAL-TIME SPEECH TRANSCRIPTION
   Phone user's speech transcribed live using Whisper or Vosk (self-hosted,
   free). Feeds into the existing EcoSkiller analysis pipeline.
   [Core Feature]

4. FULL GD ANALYSIS — SAME AS APP USERS
   Phone user gets same AI analysis: participation score, speaking time, topic
   relevance, language quality, interruption count, confidence.
   [Core Feature]

5. REPORT VIA SMS / WHATSAPP
   After GD ends, phone user receives their score via SMS (₹0.08) or WhatsApp
   (free first 1000/month via META). Works on any phone.
   [Core Feature]

6. APP USERS SEE PHONE USER LIVE
   "📞 Phone User joined" notification shown to app users. Speaking indicator
   when phone user talks. Live transcript in sidebar. Full visibility both ways.
   [Core Feature]

--- ADVANCED CAPABILITIES (Production Polish) ---

7. SPEAKER DIARIZATION
   Identifies "who said what" even if multiple people speak from one phone.
   Uses Pyannote Audio (open source). Each speaker analyzed separately.
   [Advanced]

8. MODERATOR CONTROLS
   Mute/unmute phone participants from the app. Remove disruptive callers.
   FreeSWITCH conference controls exposed via API.
   [Advanced]

9. PUSH-TO-TALK MODE
   Phone user presses * key to request speaking turn. Prevents chaotic
   overlapping in structured GD formats.
   [Advanced]

10. SPEAKING TIME ENFORCER
    Audio cue plays when phone user speaks too long ("30 seconds remaining").
    Equal time for all — even phone users.
    [Advanced]

11. AUTO-RECONNECT ON CALL DROP
    If rural user's call drops (common on 2G), system auto-calls them back.
    No need to re-dial and re-enter PIN.
    [Advanced]

12. SESSION RECORDING
    Entire GD including phone users recorded on server. App users can replay
    later. Phone users request audio link via SMS.
    [Advanced]

SECTION 04 — WHICH STT ENGINE TO USE: FULL COMPARISON
================================================================================

OPTION 1: VOSK — vosk-model-hi  [RECOMMENDED FOR MVP]
  Streaming-first, 50MB Hindi model, runs on any CPU. Fastest real-time
  response (~50ms). Designed for telephony.
  Pros:
    - Streaming API — no chunking needed
    - Runs on cheapest VPS (1 vCPU, 1GB RAM)
    - Dedicated Hindi model available
    - Used in call centers worldwide
  Cons:
    - Less accurate on Hinglish, heavy accents, noisy PSTN audio

OPTION 2: WHISPER.CPP — small model  [FOR QUALITY]
  OpenAI's Whisper in C++. Best accuracy for Hindi + Hinglish. Runs on CPU.
  MIT license.
  Pros:
    - Handles noisy 8kHz PSTN audio well
    - Robust on UP, Bihar, MP, Rajasthan accents
    - 99 languages — handles Hinglish perfectly
    - Small model: 244MB, 2-3s latency
  Cons:
    - Not truly streaming — processes in 2-3s chunks

OPTION 3: MISTRAL VOXTRAL MINI (3B)  [BEST IN 2025]
  New in 2025. Apache 2.0. Outperforms Whisper Large V3 AND GPT-4o mini
  Transcribe on Hindi benchmarks.
  Pros:
    - Built-in language understanding, not just STT
    - Can answer: "what was speaker's main argument?"
    - Apache 2.0 — completely free, commercial use OK
    - 3B model runs on CPU (needs 4-8GB RAM)
  Cons:
    - Higher RAM requirements, higher server cost

OPTION 4: FASTER-WHISPER  [BALANCED]
  Community-optimized Whisper via CTranslate2. 4x faster than original with
  same accuracy.
  Pros:
    - 4x faster, 2-3x lower memory than Whisper
    - Real-time friendly with chunked streaming
    - Same accuracy as Whisper large
    - Most used in production 2025
  Cons:
    - Still 2s window, not true streaming

RECOMMENDATION BY STAGE:
  🚀 MVP     → Vosk — fast, lightweight, 1 week to integrate
  📈 Growth  → Faster-Whisper medium — better quality, ₹800/mo VPS
  🎯 Scale   → Mistral Voxtral Mini — best accuracy + semantic understanding

SECTION 09 — SECURITY & PIN PROTECTION  [NEW — CRITICAL FOR PRODUCTION]
================================================================================

⚠ MISSING FROM PREVIOUS REPORT — CRITICAL FOR PRODUCTION

A 6-digit PIN has only 1,000,000 combinations. Without protection, a bad actor
can brute-force their way into any live GD session in minutes. This section
covers every security layer needed before going live.

THREAT 1: PIN BRUTE-FORCE ATTACK
  Risk: Attacker scripts a call that tries 000000, 000001, 000002... until they
  find a valid session PIN and join someone else's GD.
  Solution — 3 layers of protection:
    - Rate limit: Max 3 wrong PIN attempts per phone number per 10 minutes,
      then block the number for 30 minutes
    - Lockout: After 5 total wrong attempts on a session PIN, disable that PIN
      entirely and notify the session host
    - PIN expiry: Session PIN expires automatically 5 minutes after session ends

THREAT 2: SESSION HIJACKING VIA CALLER ID SPOOFING
  Solution:
    - Never rely on Caller ID as the only auth — always require PIN
    - Use PIN + phone number whitelist (optional) for high-security sessions
    - Log all joining attempts with timestamp + Caller ID + IP of FreeSWITCH
      gateway

THREAT 3: STALE PINs BEING REUSED
  Risk: A PIN from a GD session 2 days ago still works and someone accidentally
  joins a new session that reused the same PIN.
  Solution:
    - Generate PINs with UUID-based seed, not sequential numbers
    - Hard expiry: PIN becomes invalid when session status = "ended"
    - Same 6 digits cannot repeat within 24 hours

THREAT 4: SPAM / ABUSE CALLS
  Risk: Someone repeatedly dials your conference number to waste SIP trunk
  minutes (costs you money) or disrupt a live session.
  Solution:
    - Global rate limit: max 10 calls per hour from same number to your service
    - Session max capacity: define max phone participants per session (e.g., 3)
    - Moderator can hard-ban a phone number from the app UI
    - Auto-hangup after 3 wrong PINs from same call

SECURITY IMPLEMENTATION CHECKLIST:
  ✓ Rate limit: 3 wrong PINs per number per 10 min
  ✓ Session lockout after 5 total wrong attempts
  ✓ PIN auto-expires when session ends
  ✓ No sequential PINs — use cryptographically random
  ✓ Log all join attempts (number, time, success/fail)
  ✓ Max phone participants cap per session
  ✓ Moderator can ban phone numbers in real-time
  ✓ Auto-hangup on repeated wrong PIN from same call

SECTION 11 — LATENCY & AUDIO SYNC PROBLEM  [NEW]
================================================================================

⚠ MISSING FROM PREVIOUS REPORT — REAL PRODUCTION ISSUE

PSTN calls introduce a natural 200–400ms delay compared to WebRTC's ~50ms. This
means the phone user hears app users slightly delayed, and vice versa. Without
managing this, users will experience echo, double-talk, and awkward overlaps.

LATENCY COMPARISON:
  WebRTC Users         : ~50ms  (Near-instantaneous, best case)
  PSTN Phone User      : 200–400ms (PSTN encoding + SIP processing + FS mixing)
  STT Processing       : +50ms – 2s (Vosk: +50ms | Whisper: +2-3s chunks)
                         Note: Parallel to audio — doesn't block the call

FIX 1: ECHO CANCELLATION (Critical)
  Without echo cancellation, the phone user will hear their own voice echoed
  back ~400ms later. This is highly disruptive.
  - Enable FreeSWITCH mod_echo_cancellation (built-in, free, one config line)
  - For BSNL calls (known echo issues), add aggressive echo suppression level 3
  - FreeSWITCH conference does AEC by default when configured correctly

FIX 2: JITTER BUFFER MANAGEMENT
  Rural 2G calls have inconsistent packet timing (jitter). Without a buffer,
  audio sounds choppy and broken to all participants.
  - Configure FreeSWITCH jitter buffer: jitter-buffer-msec=60
  - Set adaptive buffer that increases automatically if packet loss detected
  - Adds ~60ms latency but gives smooth audio even on unstable 2G signals

FIX 3: AUDIO QUALITY UPSAMPLING
  Phone user audio comes in at 8kHz. App users are at 48kHz. When mixed, the
  phone user sounds "muffled" compared to everyone else.
  - FreeSWITCH automatically resamples 8kHz → 48kHz during conference mixing
  - Enable mod_samplerate with Speex resample algorithm (better quality)
  - Result: phone user's voice sounds cleaner to app users

FIX 4: DOUBLE-TALK PREVENTION
  Because of the 200-400ms delay gap, phone users often accidentally interrupt
  because they can't hear that someone already started speaking.
  - In structured GDs: use push-to-talk for phone users (press * to speak)
  - In free-form GDs: add a 200ms "hold-off" before phone user audio unmutes
  - Show app users a "⚠️ Phone user may hear you delayed" notice in UI

REALISTIC EXPECTATION:
  With all fixes applied, phone user experience will be similar to a
  speakerphone on a Group Call. Perfectly usable for GD. The 200–400ms delay
  is perceptible but not conversation-breaking. Most rural users are already
  accustomed to this from normal conference calls on BSNL/MTNL.

SECTION 13 — SCORE FAIRNESS: 8kHz vs 48kHz PROBLEM  [NEW]
================================================================================

⚠ MISSING FROM PREVIOUS REPORT — AFFECTS CORE PRODUCT FAIRNESS

Phone users call in at 8kHz audio (PSTN narrowband). App users stream at 48kHz
(WebRTC wideband). This quality difference directly causes STT to produce
lower-quality transcripts for phone users — fewer words recognized, more errors
— which means the AI will unfairly score phone users lower. This is a product
integrity issue.

WITHOUT FAIRNESS CORRECTION:
  - Phone user says "climate change mitigation" — STT hears "climate hang
    mitigation" (8kHz garble)
  - STT Word Error Rate: ~15-20% for phone users vs ~3-5% for app users
  - AI penalizes phone user for "lower quality speech" when it's actually
    audio quality degradation
  - Phone user gets 58/100 score; same performance app user gets 75/100
  - Rural students disadvantaged purely because of infrastructure

WITH FAIRNESS CORRECTION APPLIED:
  - System detects participant type: phone vs app at join time
  - Apply "phone participant" flag to all their transcription data
  - STT confidence threshold lowered for phone users (accept lower-confidence
    words)
  - Apply +10–15% score normalization for known audio quality disadvantage
  - Show "📞 Joined via phone call" note in their report so evaluators are aware

TECHNICAL FIXES FOR SCORE FAIRNESS:

  1. FreeSWITCH Preprocessing
     Apply wideband simulation (bandwidth extension) before sending 8kHz audio
     to STT. mod_opus can upsample PSTN audio. Adds ~20ms latency but
     significantly improves STT accuracy.

  2. Score Normalization Algorithm
     Track historical STT accuracy per connection type. If phone users
     consistently get 15% lower STT word confidence, apply 15% upward
     normalization to their linguistic quality score. Recalibrate monthly.

  3. Transparent Disclosure
     Show in report: "Score calculated with phone-call audio adjustment
     applied." Both the student and evaluator know the context. Builds trust
     rather than hiding the normalization.

IMPORTANT: Score normalization must be based on measured data, not a hardcoded
offset. Measure actual STT error rates on 100+ real phone calls from Indian
telecom networks, then derive the correction factor empirically. This makes the
scoring defensible to users.

SECTION 18 — WEBRTC ↔ FREESWITCH BRIDGE INTEGRATION  [NEW]
================================================================================

⚠ INCOMPLETE IN PREVIOUS REPORT — THE HARDEST TECHNICAL PART

The original report explained how phone users join via FreeSWITCH, and
separately how app users use WebRTC — but never explained how these two systems
are connected to each other. This is actually the hardest integration step.

THE PROBLEM:
  Your EcoSkiller app users are in a WebRTC room (Daily.co, Agora, or your own
  WebRTC server). Phone users join a FreeSWITCH conference room. These are two
  completely different audio systems. Without an explicit bridge, app users and
  phone users are in entirely separate audio sessions and cannot hear each other.

--- 3 ARCHITECTURE OPTIONS — CHOOSE ONE ---

OPTION A: FULL FREESWITCH (Simplest) — RECOMMENDED FOR LONG-TERM
  Idea: Migrate ALL participants (app users too) to FreeSWITCH conference.
  App users connect via WebRTC-SIP using their browser. Phone users connect via
  PSTN-SIP. Everyone is in the same FreeSWITCH conference room — no bridge.
  Advantages:
    - Simplest architecture — one system for everyone
    - No bridge to maintain or troubleshoot
    - FreeSWITCH natively supports WebRTC via WebSocket SIP
    - Works with standard SIP.js in browser (open source)
  Disadvantages:
    - Must migrate existing WebRTC infrastructure
    - Lose features from Daily.co/Agora (video, recording, etc.)
    - FreeSWITCH WebRTC setup is complex — requires STUN/TURN config
  Best for: Long-term scale

OPTION B: SIP GATEWAY BRIDGE (Balanced) — GOOD FOR EXISTING APPS
  Idea: Keep existing WebRTC (Daily.co/Agora/Jitsi) for app users. Run
  FreeSWITCH separately for phone users. Use a SIP-WebRTC gateway to bridge the
  two. Audio from FreeSWITCH forwarded to WebRTC room and vice versa.
  Architecture:
    📱 App Users (WebRTC) ⇄ 🔗 SIP Gateway (RTPengine/Bridgit) ⇄ ⚙️ FreeSWITCH ⇄ ☎️ Phone Users
  Advantages:
    - App users stay on existing WebRTC — no migration needed
    - RTPengine (open source, Kamailio project) handles SIP↔WebRTC bridge
  Disadvantages:
    - Higher latency (~100ms extra hop through gateway)
    - More moving parts to configure and maintain
  Best for: Existing WebRTC apps

OPTION C: TWILIO AS BRIDGE ONLY (Fastest to Build) — HYBRID
  Idea: Use Twilio Conference specifically as the audio bridge between PSTN
  and WebRTC. App users connect to Twilio Conference using Twilio SDK. Phone
  users dial in via Twilio PSTN. Self-hosted Whisper still handles STT.
  Only Twilio Conference bridging is paid.
  Advantages:
    - Easiest and fastest to implement (1-2 days)
    - Twilio handles all WebRTC↔PSTN complexity
    - Battle-tested, production-ready immediately
    - Use free self-hosted Whisper for STT to cut cost
  Disadvantages:
    - Twilio Conference: ₹0.15/min per participant (app + phone)
    - Dependency on Twilio vendor — not 100% open source
    - Higher cost at scale
  Best for: MVP validation. Move to Option A or B after product-market fit.

DECISION MATRIX:
  Criteria              | Option A (Full FreeSWITCH) | Option B (SIP Gateway) | Option C (Twilio)
  ----------------------|---------------------------|------------------------|------------------
