## 8️⃣ EVIDENCE PRESERVATION ARCHITECTURE

### Evidence Chain Integrity

```
EVIDENCE_PRESERVATION_PROTOCOL:

Step 1 — CAPTURE (at signal receipt, before any action)
  Capture all relevant data points at moment of signal:
  - Communication logs (platform-internal only, legally authorized scope)
  - Session recording reference (if Dojo match or live session)
  - Content references (images, files, posts, comments)
  - Access logs (timestamps, IP hash, device fingerprint hash)
  - Actor profile snapshots (frozen at incident moment)
  - Platform state at time of incident

Step 2 — HASH (immediately after capture)
  SHA-256 hash each evidence item independently
  Build SHA-256 Merkle tree across all evidence items
  Store Merkle root as evidence_hash_root in output
  This makes any tampering with any single evidence item detectable

Step 3 — SEAL (append-only write to LEGAL_EVIDENCE_VAULT_AGENT)
  Write sealed evidence package to LEGAL_EVIDENCE_VAULT_AGENT
  Evidence vault is:
  - Append-only (no modifications ever)
  - Encrypted at rest (AES-256)
  - Access restricted to: CHILD_SAFETY_OFFICER, PLATFORM_ADMIN, Law Enforcement
    (via formal legal request process only)
  - Geographically replicated to comply with data sovereignty rules
  - Retained for minimum: 10 years or duration of any open legal proceeding
    (whichever is longer)

Step 4 — CHAIN OF CUSTODY LOG
  Every access to the evidence package is logged:
  - accessor_id
  - accessor_role
  - access_timestamp_utc
  - access_purpose
  - access_authorization_reference
  Chain of custody log is itself append-only and tamper-evident

Step 5 — LEGAL ADMISSIBILITY CONFIRMATION
  Evidence package receives legally_admissible = true when:
  - All hash verifications pass
  - Chain of custody is unbroken
  - Evidence was captured before any protective action modified platform state
  - Evidence package is complete (no missing mandatory items)
```

### Evidence Item Classification

```
TIER_1_EVIDENCE (Mandatory capture for all Category A incidents):
  - Full communication thread (platform-internal) with actor IDs and timestamps
  - Session recording reference and access token (time-locked)
  - Offending content hash and reference (not raw content in standard log)
  - Actor profile snapshots (both minor and adult actor at incident moment)
  - IP hash and device fingerprint hash (privacy-preserving, not raw IP)
  - Platform audit trail entries for past 72 hours for both actors

TIER_2_EVIDENCE (Mandatory capture for Category B incidents):
  - Relevant communication excerpts (platform-internal)
  - Content references involved
  - Actor profile snapshots
  - Recent access log summary (past 24 hours)

TIER_3_EVIDENCE (Standard capture for Category C incidents):
  - Signal payload
  - Actor identifiers
  - Platform surface context
  - Timestamp record
```
