## PREAMBLE — ANTIGRAVITY DEFINED

In the Ecoskiller ecosystem, **Antigravity** is defined as:

> The structural force that prevents the platform from collapsing under the weight of disputes, scoring anomalies, scoring manipulation, mentor misconduct, royalty fraud, idea theft claims, institutional escalations, billing conflicts, GD integrity violations, Dojo scoring challenges, and trust breakdowns between any two parties.

Without the CONFLICT_RESOLUTION_AGENT (CRA), every one of the 40–45 moving system components becomes a gravity source — pulling trust, reliability, and platform credibility downward.

The CRA is the **only system authorized** to:
- Accept, classify, route, adjudicate, and close conflict records
- Apply trust penalties, reputation corrections, and score reversals
- Issue immutable resolution verdicts with audit-chain proof
- Trigger cross-service cascade corrections across 15+ microservices
- Report systemic friction patterns to the Admin Governance Layer

The CRA does not interpret. It enforces.

```
   │

   │
   ├─── [Party Appeals] ──→ APPEAL_OPENED
   │                                  │
   │                     ┌────────────┘
   │                     ▼
   │             APPEAL_ADJUDICATED
   │                     │
   │                     ▼
   │               FINAL_VERDICT
   │
   └─── [Timeout] ──────→ CLOSED_ABANDONED

```

**State Transition Rules (LOCKED):**

- SUBMITTED → CLASSIFIED: ≤ 2 minutes (automated classifier)
- CLASSIFIED → ROUTED: ≤ 5 minutes (deterministic router)
- ROUTED → AUTO_ADJUDICATED: ≤ 30 minutes if rule-resolvable
- ROUTED → UNDER_REVIEW: ≤ 24 hours SLA for human reviewer assignment
- UNDER_REVIEW → VERDICT_ISSUED: ≤ 72 hours SLA
- VERDICT_ISSUED → APPEAL_OPENED: ≤ 7 days window
- APPEAL_ADJUDICATED: ≤ 96 hours
- CLOSED_ABANDONED: triggered if no response after 14 days

No state may be skipped.
No state may be reversed except via APPEAL_OPENED.

## SECTION IV — CONFLICT CLASSIFIER ENGINE (SEALED)

The Classifier is the first layer the CRA runs upon conflict submission.
It is **deterministic** — no ML, no inference.

### Classification Algorithm

```python
def classify_conflict(submission: ConflictSubmission) -> ConflictRecord:

    # Step 1: Domain mapping from subject entity type
    domain_id = DOMAIN_MAP[submission.subject_entity_type]
    # e.g. 'gd_session' → 'CD-08', 'invoice' → 'CD-04', 'idea' → 'CD-05'

    # Step 2: Subtype resolution from trigger signals
    subtype_id = SUBTYPE_RESOLVER[domain_id][submission.trigger_signals]
    # trigger_signals is a structured dict from submitting service

    # Step 3: Severity assignment
    severity = SEVERITY_MATRIX[domain_id][subtype_id][submission.affected_user_count]

    # Step 4: Auto-resolvability check
    routing_rule = get_routing_rule(domain_id, subtype_id)
    auto_resolvable = routing_rule.auto_resolvable

    # Step 5: SLA assignment
    sla_hours = routing_rule.sla_hours_auto if auto_resolvable else routing_rule.sla_hours_human
    sla_deadline = now() + timedelta(hours=sla_hours)

    return ConflictRecord(
        domain_id=domain_id,
        subtype_id=subtype_id,
        severity_level=severity,
        auto_resolvable=auto_resolvable,
        sla_deadline=sla_deadline,
        current_state='CLASSIFIED'
    )
```

### RESOLUTION PROTOCOLS (LOCKED)

#### SC-01 · GD Score Anomaly

```
PROTOCOL GD-SCORE-VERIFY:

1. Retrieve session logs from Redis (session_id)
2. Retrieve state_machine_transitions log from PostgreSQL
3. Replay speaking_token sequence against participation_log
4. IF computed_score ≠ stored_score:
   → generate correction_record: SCORE_REVERSAL
   → emit: gd.score.corrected event
   → notify: both participant and scoring service
5. IF computed_score = stored_score:
   → verdict: DISMISSED
   → reason: "Logs confirm score accuracy"
6. Emit: conflict.resolved event
7. Write immutable verdict hash
```

#### SC-06 · Peer Score Collusion Detection

```
PROTOCOL COLLUSION-DETECT:

1. Retrieve peer_scores for match_id
2. Run variance_check: IF stddev(all_peer_scores) < COLLUSION_THRESHOLD:
   → flag: COLLUSION_SUSPECTED
3. Retrieve network metadata: IPs, device fingerprints, timestamps
4. IF ≥2 raters share same IP or device within 60 seconds:
   → verdict: UPHELD
   → correction: nullify colluding peer scores
   → recompute: average from remaining valid scores
   → penalty: TRUST_SCORE_DEDUCTION (both colluding accounts)
5. IF no network overlap found:
   → verdict: DISMISSED
   → recommendation: Flag for manual audit if pattern recurs
```

#### BR-01 · Invoice Amount Dispute

```
PROTOCOL INVOICE-RECOMPUTE:

1. Retrieve invoice_id from conflict record
2. Pull: plan_type, usage_events, billing_period from Billing Service
3. Recompute: invoice_total using canonical pricing formula
4. Compare: computed_total vs stored_total
5. IF diff > ₹0.01:
   → verdict: UPHELD
   → correction: PENALTY_CREDIT for diff amount to user wallet
   → re-issue: corrected invoice with correction_ref
6. IF diff = 0:
   → verdict: DISMISSED
   → explanation: "Invoice recomputed; original amount confirmed accurate"
7. Write audit log entry
```

#### BR-06 · Double Charge Detection

```
PROTOCOL DOUBLE-CHARGE-VERIFY:

1. Retrieve all payment_gateway_events for user_id within 24-hour window
2. Deduplicate by: amount + gateway_order_id + timestamp_window(60s)
3. IF duplicate found:
   → verdict: UPHELD
   → correction: FULL_REFUND of duplicate charge
   → notify: payment gateway webhook for reversal
   → emit: billing.refund.issued
4. IF no duplicate:
   → verdict: DISMISSED
```

#### IO-01 · Idea Similarity Dispute

```
PROTOCOL SIMILARITY-ADJUDICATE:

1. Retrieve both idea records from Idea Registry
2. Pull similarity_score from Idea Similarity Engine
3. Pull timestamp_proof from Immutable Archive
4. Compare: created_at timestamps
5. IF similarity ≥ 85% AND timestamps differ:
   → party with EARLIER timestamp: PROTECTED
   → party with LATER timestamp: WARNING issued
   → IF later idea was commercialized: FLAG for CD-05/IO-03
6. IF similarity ≥ 85% AND timestamps within 24h:
   → verdict: PARTIAL (both claim simultaneous origin)
   → recommendation: Human review required
7. Write fingerprint audit trail
```

#### MS-01 · GD Network Drop Dispute

```
PROTOCOL SESSION-INTEGRITY-VERIFY:

1. Pull session_logs from Jitsi Videobridge for room_id
2. Pull WebRTC ICE connectivity events for participant_id
3. Cross-reference: drop_timestamp vs speaking_token_active window
4. IF drop occurred OUTSIDE active token window:
   → no score impact; verdict: NO_ACTION
5. IF drop occurred DURING active token window:
   → check: was token_skipped already recorded?
   → IF skipped AND system-confirmed drop: restore token credit
   → verdict: UPHELD (partial restoration)
6. IF participant's network logs show self-disconnection:
   → verdict: DISMISSED
```

## SECTION VI — HUMAN REVIEW ROUTER (SEALED)

Conflicts not auto-resolvable are routed to human reviewers via the **Reviewer Assignment Engine (RAE)**.

### Assignment Algorithm

```python
def assign_reviewer(conflict: ConflictRecord) -> Reviewer:

    # Step 1: Get eligible reviewers by domain expertise
    candidates = Reviewer.query(
        domain_expertise=conflict.domain_id,
        available=True,
        current_load__lt=MAX_CONCURRENT_CASES
    )

    # Step 2: Exclude reviewers with prior relationship to parties
    candidates = [r for r in candidates
                  if not has_prior_interaction(r, conflict.initiator_user_id)
                  and not has_prior_interaction(r, conflict.respondent_user_id)]

    # Step 3: Score by workload + expertise depth
    ranked = sorted(candidates,
                    key=lambda r: (r.expertise_depth[conflict.domain_id] * 0.6)
                                - (r.current_load * 0.4),
                    reverse=True)

    # Step 4: Assign top candidate
    reviewer = ranked[0]
    reviewer.current_load += 1
    return reviewer

    # Step 5: If no eligible reviewer found
    # → escalate to Senior Moderator pool
    # → SLA extends by +24h
    # → emit: reviewer.assignment.escalated
```

### Reviewer Recusal Rules (LOCKED)

- A reviewer must recuse if they have engaged with either party in any context within 90 days
- A reviewer must recuse if they are a member of the same institution or company tenant as either party
- Recusal automatically triggers re-assignment
- No reviewer may act on more than 3 CRITICAL severity cases simultaneously

### Cascade Execution Protocol

```python
def execute_corrections(verdict: Verdict):

    corrections = verdict.corrections_issued  # list of ConflictCorrection

    # Phase 1: Pre-validation
    for correction in corrections:
        validate_target_exists(correction.target_entity_id, correction.target_service)
        if not validate_target_exists():
            → STOP cascade for this correction
            → log: CORRECTION_TARGET_MISSING
            → alert: Admin Governance Service

    # Phase 2: Atomic execution within service
    for correction in corrections:
        try:
            result = call_service(
                correction.target_service,
                correction.correction_type,
                correction.target_entity_id,
                correction.after_value
            )
            mark_correction_applied(correction.correction_id, result)
            emit_event(f"correction.{correction.correction_type.lower()}.applied",
                       correction.to_event_payload())
        except ServiceError as e:
            → log: CORRECTION_EXECUTION_FAILED
            → emit: correction.failed event
            → retry with exponential backoff (max 3 attempts)
            → if still failed: escalate to Admin Governance

    # Phase 3: Cross-service reconciliation
    emit_event("conflict.verdict.corrections_complete", {
        "conflict_id": verdict.conflict_id,
        "corrections_count": len(corrections),
        "completed_at": now()
    })

    # Phase 4: Immutable hash-chain write
    write_to_immutable_archive(
        record_type="CONFLICT_VERDICT_EXECUTION",
        payload=verdict.to_archive_payload(),
        retention_years=15
    )
```
