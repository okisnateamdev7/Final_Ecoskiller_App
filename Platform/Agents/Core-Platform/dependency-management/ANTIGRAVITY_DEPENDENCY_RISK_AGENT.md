## SECTION 4 — DEPENDENCY RISK TAXONOMY (SEALED)

All dependency risk items must be classified into one of these 7 risk categories. No item may be left unclassified.

### Category 1: CVE & VULNERABILITY RISK (CVR)
Active or known security vulnerabilities in a dependency. Governed by VULN-I, VULN-Q, Section 151.10.

| Sub-type | Code | Description |
|---|---|---|
| Critical CVE | CVR-C | CVSS ≥ 9.0 — deployment blocked |
| High CVE | CVR-H | CVSS 7.0–8.9 — 72h SLA |
| Medium CVE | CVR-M | CVSS 4.0–6.9 — 14-day SLA |
| Low CVE | CVR-L | CVSS < 4.0 — 30-day SLA |
| Unscanned dependency | CVR-U | Dependency not in CVE scan results |
| Zero-day disclosed | CVR-Z | Public zero-day with no patch yet |

### Category 2: LICENSE COMPLIANCE RISK (LCR)
License terms that conflict with commercial SaaS operation, distribution, or platform goals.

| Sub-type | Code | Description |
|---|---|---|
| Copyleft infection risk | LCR-CL | GPL/AGPL dependency that may require open-sourcing SaaS platform |
| License mismatch | LCR-LM | Dependency license incompatible with commercial distribution |
| Undeclared license | LCR-UD | Dependency has no license declared |
| License change | LCR-LC | Vendor changed license post-adoption (e.g., BSL, SSPL) |
| Dual-license risk | LCR-DL | Dependency with commercial license requirement for SaaS use |
| Patent grant absent | LCR-PG | Dependency lacks patent grant in license (risk of patent claim) |

### Category 3: VENDOR LOCK-IN RISK (VLR)
Architectural or contractual binding to a single vendor that makes exit technically infeasible or commercially punitive. Governed by Vendor Lock-In Law Sections A–J, R56.11.

| Sub-type | Code | Description |
|---|---|---|
| No exit strategy | VLR-NE | Integration without documented exit plan (R56.11 violation) |
| Proprietary data format | VLR-PD | Data stored in non-portable, vendor-specific format |
| SDK direct coupling | VLR-SC | Business logic directly coupled to vendor SDK (no abstraction layer) |
| Single-vendor critical path | VLR-SV | Critical system path has no documented fallback vendor |
| Commercial lock-in | VLR-CM | Contract terms block exit (punitive egress, exclusivity, no migration help) |
| Domain cross-contamination | VLR-DX | Vendor decision in one domain forces same vendor in another (Dojo Law Section D) |
| Proprietary API binding | VLR-PA | Core logic bound to proprietary API with no standard equivalent |
| Cloud provider binding | VLR-CB | Infrastructure tied to single cloud provider with no IaC abstraction |

### Category 4: SUPPLY CHAIN RISK (SCR)
Risks from upstream dependency compromise, unsigned artifacts, unverified build chains. Governed by Section 151.10, VULN-I, CD-D.

| Sub-type | Code | Description |
|---|---|---|
| Unsigned artifact | SCR-UA | Container image or package without SHA256 / signed verification |
| Unverified build chain | SCR-UB | Dependency sourced from unofficial or untrusted registry |
| Typosquatting risk | SCR-TY | Package name similar to known popular package — possible typosquat |
| Transitive CVE | SCR-TC | Vulnerability in an indirect (transitive) dependency |
| Missing SBOM entry | SCR-MS | Dependency present in artifact but absent from SBOM |
| Compromised maintainer risk | SCR-CM | Dependency maintainer has known past compromise incident |
| Abandoned package | SCR-AP | Dependency has no commits/releases in 18+ months |

### Category 5: VERSION & UPGRADE RISK (VUR)
Risks from pinned versions, missing upgrades, breaking changes, and end-of-life. Governed by Rule 408.3 (dependency upgrade safety).

| Sub-type | Code | Description |
|---|---|---|
| Unpinned version | VUR-UP | Dependency without locked version (floating `latest` or `*`) |
| End-of-life version | VUR-EL | Dependency using a version past vendor's EOL date |
| Breaking change pending | VUR-BP | Current version has known breaking change in next major release |
| Major version lag | VUR-ML | Dependency 2+ major versions behind current stable |
| Minor version lag | VUR-MV | Dependency 1 major version behind current stable |
| Patch lag | VUR-PL | Security patch available but not applied |
| No upgrade path | VUR-NP | Dependency has no documented migration path to next version |

### Category 6: AVAILABILITY & CONTINUITY RISK (ACR)
Risks from single points of failure, vendor sunset, project abandonment, or critical path dependency without fallback.

| Sub-type | Code | Description |
|---|---|---|
| Single point of failure | ACR-SF | Critical dependency with no redundancy or fallback |
| Vendor sunset risk | ACR-VS | Vendor has signaled EOL, acquisition, or shutdown |
| Abandoned project | ACR-AP | Open-source project with no active maintainer |
| Missing circuit breaker | ACR-CB | External dependency called without circuit breaker pattern |
| No health monitoring | ACR-HM | External dependency without health/latency monitoring |
| No fallback implementation | ACR-NF | Critical feature depends on single external service with no fallback |
| Data egress risk | ACR-DE | Vendor charges prohibitive egress fees making exit costly |

### Category 7: COMPLIANCE & REGULATORY RISK (CRR)
Risks where a dependency conflicts with GDPR, DPDP, data residency, or audit requirements. Governed by Rule 410.

| Sub-type | Code | Description |
|---|---|---|
| Data residency violation | CRR-DR | Dependency sends or stores data outside required jurisdiction |
| PII logging risk | CRR-PL | Dependency logs PII in its default configuration |
| Audit trail absence | CRR-AT | Vendor integration provides no audit log of data access |
| Minor data exposure | CRR-MD | Dependency may process or retain data from minors without consent controls |
| Regulatory incompatibility | CRR-RI | Dependency not compliant with GDPR / India DPDP patterns required by platform |
| Third-party DPA absent | CRR-DP | No Data Processing Agreement in place for data-handling vendor |
| Vendor compliance unverified | CRR-UV | Vendor security posture and compliance certifications not assessed |

### DIMENSION DR-1: SBOM COMPLETENESS & ACCURACY
**Laws:** VULN-I, Section 151.10, CD-D | **Weight:** 10%

The Software Bill of Materials (SBOM) is the foundational artifact for all dependency risk work. A missing or incomplete SBOM is a pre-condition failure.

**DR-1.1 — SBOM existence**
- SBOM file must exist at `/sbom/sbom.json` or `/sbom/sbom.spdx` in SPDX 2.3 or CycloneDX 1.4+ format
- Absence: immediate **DRA_BLOCK** — no other scan can proceed reliably

**DR-1.2 — SBOM coverage**
For each service or artifact layer:
- Backend (Python): every entry in `requirements.txt` / `pyproject.toml` + all transitive deps declared
- Frontend Flutter: every entry in `pubspec.yaml` + all transitive deps
- Frontend Next.js: every entry in `package.json` + `package-lock.json` / `yarn.lock`
- Infra: all Helm chart dependencies, all Docker base images with digests
- CI/CD: all GitHub Actions / GitLab CI step dependencies declared
- AI/ML: all model sources, APIs, frameworks declared with version and cost tier (R28)
Missing any layer: HIGH RISK per layer

**DR-1.3 — SBOM accuracy**
- Every SBOM entry must have: `name`, `version`, `purl` (Package URL), `license`, `supplier`, `checksum`
- Floating versions in SBOM (e.g., `>=1.0`) without resolved pinned version: MEDIUM RISK per entry
- Missing checksum/hash for any package: HIGH RISK per entry

**DR-1.4 — SBOM freshness**
- SBOM must be regenerated on every build (CD-D enforcement)
- SBOM older than the most recent build: HIGH RISK
- SBOM not linked to CI pipeline stage: MEDIUM RISK

**DR-1.5 — Undeclared dependency detection**
Scan all generated code for `import`, `require`, `from X import` statements and cross-reference against SBOM:
- Any imported package not in SBOM: **UNDECLARED_DEPENDENCY** — HIGH RISK per instance
- Pattern: scan all `.py`, `.dart`, `.ts`, `.js`, `.yaml`, `.tf` files

**Scoring:**
- SBOM absent: DRA_BLOCK
- 100% coverage + all fields: 10/10
- <100% coverage: score = (covered_count / total_count) × 10, floor at 2

### DIMENSION DR-2: CVE & VULNERABILITY RISK SCAN
**Laws:** VULN-A through VULN-Q, VULN-G (SLA enforcement) | **Weight:** 15%

**DR-2.1 — CVE scan execution**
Every dependency in SBOM must be checked against CVE databases (NVD, OSV, GHSA, Snyk DB). Scan must cover:
- All direct dependencies
- All transitive (indirect) dependencies
- All Docker base images (each layer)
- All CI/CD action/step dependencies
- All Helm chart sub-dependencies

**DR-2.2 — Critical CVE (CVSS ≥ 9.0) — SLA: ≤24 hours**
Any critical CVE in any dependency used in production path: **CVR-C**
- Status: `DRA_BLOCK` — deployment frozen until patched or mitigated
- Mitigation options: version upgrade, vendor patch, dependency replacement, compensating control
- Compensating control must be documented if patch not available (zero-day scenario VULN-H)

**DR-2.3 — High CVE (CVSS 7.0–8.9) — SLA: ≤72 hours**
- Any high CVE in production-path dependency: HIGH RISK
- Must be patched or mitigated within 72 hours of detection
- Unmitigated beyond 72h: escalates to `DRA_BLOCK`

**DR-2.4 — Medium CVE — SLA: ≤14 days**
- Any medium CVE: MEDIUM RISK
- Unmitigated beyond 14 days: escalates to HIGH RISK

**DR-2.5 — Transitive CVE tracking (SCR-TC)**
- Transitive dependencies with CVEs carry the same scoring rules
- "But it's a transitive dependency" is NOT an exemption
- SBOM must declare transitive chain to enable tracking

**DR-2.6 — Domain & tenant impact amplification (VULN-C)**
Standard CVE scores must be amplified for:
- Billing/payment components: CVE severity + 1 tier
- Minor-facing (student) components: **zero-tolerance** — any HIGH+ CVE = DRA_BLOCK
- Science assessment engine: CVE severity + 1 tier
- Real-time Dojo services (VULN-J): extra scrutiny on WebRTC/WebSocket CVEs

**DR-2.7 — Unscanned dependency**
Any dependency in artifacts but not covered by CVE scan: **CVR-U** — MEDIUM RISK (may hide critical CVEs)

**Scoring:**
- Any Critical CVE in prod path: DRA_BLOCK (score is irrelevant)
- Each High CVE: -2 pts
- Each Medium CVE: -0.5 pts
- Each unscanned dep: -0.5 pts
- Clean = 10/10

**DR-3.4 — Patent risk (LCR-PG)**

**DR-3.5 — Commercial license validation for SaaS**
For any dependency with `PROPRIETARY` or `COMMERCIAL` license:
- Commercial SaaS license purchased and documented: SAFE
- License absent or only personal/development license held: DRA_BLOCK

**Scoring:**

- Any proprietary dependency without commercial license: DRA_BLOCK
- Any GPL dependency in SaaS distribution path: HIGH RISK
- Each undeclared license: -0.5 pts
- Each license change (BSL/SSPL): -2 pts

### DIMENSION DR-4: VENDOR LOCK-IN RISK ASSESSMENT
**Laws:** Vendor Lock-In Law (A–J), R56.11, R56.19, R56.20, Portability Requirements | **Weight:** 10%

**DR-4.1 — Exit strategy completeness (R56.11)**
Every third-party integration and critical dependency must have an exit plan. Scan `/docs/integrations/` or `/config/integration-registry.yaml` for:
- Vendor replacement approach declared
- Data portability steps documented
- Decommission checklist present
- Kill-switch capability implemented (R39, R40)
Missing exit plan for any integration: `VLR-NE` — HIGH RISK. "No exit strategy → INTEGRATION FORBIDDEN."

**DR-4.2 — Abstraction layer presence (VLR-SC)**
Business logic must not be directly coupled to vendor SDKs. For each critical dependency:
- Payment (Razorpay/Stripe): `PaymentGatewayInterface` abstraction required — direct SDK call in business logic: HIGH RISK
- Email (SendGrid/Postfix): `EmailProviderInterface` required — hardcoded provider SDK: MEDIUM RISK
- SMS (Twilio/Jasmin): `SMSProviderInterface` required
- Video (LiveKit/Jitsi): `VideoRoomInterface` required
- LLM API (OpenAI/Anthropic): `LLMProviderInterface` required — direct API call in domain logic: HIGH RISK
- Cloud storage (MinIO/S3): `ObjectStorageInterface` required
- Search (OpenSearch): `SearchEngineInterface` required

**DR-4.3 — Lock-in category assessment (Vendor Lock-In Law Section B)**
Assess all 7 lock-in categories:
1. **Infrastructure lock-in**: Is infra IaC (OpenTofu) truly provider-agnostic? Any AWS/GCP/Azure-specific primitives in OpenTofu without abstraction: HIGH RISK
2. **Data lock-in**: Data in non-portable format (proprietary binary, vendor-specific schema): HIGH RISK per data store
3. **Identity/Security lock-in**: Keycloak OSS — portable. Any proprietary IAM without migration path: HIGH RISK
4. **Messaging/Events lock-in**: Kafka OSS — portable. Any managed-Kafka with vendor-specific APIs: MEDIUM RISK
5. **CI/CD lock-in**: GitLab CI — portable OSS. Any proprietary CI without export format: MEDIUM RISK
6. **Monitoring lock-in**: Prometheus/Grafana/Loki — all OSS, portable. Any proprietary observability: MEDIUM RISK

### DIMENSION DR-5: VERSION CURRENCY & UPGRADE RISK
**Laws:** Rule 408.3 (dependency upgrade safety), VULN-G | **Weight:** 8%

**DR-5.1 — Pinned version audit (VUR-UP)**
Scan all dependency manifests:
- `requirements.txt`: any `package>=x.y` without upper bound or exact pin: HIGH RISK (production must use exact pins)
- `pubspec.yaml`: any `^x.y.z` or `>=x.y.z` without exact version in `pubspec.lock`: MEDIUM RISK
- `package.json`: any `^x.y.z` without resolved lock file: MEDIUM RISK
- Docker images: any `FROM image:latest` or `FROM image:major.minor` without digest: HIGH RISK
- Helm charts: any `version: >=x.y` without pinned chart version: MEDIUM RISK
All production-bound dependencies must use exact pinned versions.

**DR-5.2 — End-of-life detection (VUR-EL)**
Check each dependency version against known EOL data:
- Python 3.11: supported until Oct 2027 — SAFE
- Python 3.9 or below: EOL — CRITICAL RISK (must upgrade)
- Any PostgreSQL < 13: EOL — HIGH RISK
- Any Redis < 6: EOL — HIGH RISK
- Any Flutter < 3.0: MEDIUM RISK (check EOL date)
- Any Node.js LTS version EOL: HIGH RISK
Flag all EOL dependencies with exact EOL date and upgrade target.

**DR-5.3 — Major version lag (VUR-ML)**
For each dependency, compare used version to current stable:
- 2+ major versions behind: HIGH RISK
- 1 major version behind: MEDIUM RISK
- On current major but behind on minor/patch: LOW RISK
Priority: security-critical dependencies (auth, crypto, gateway) must be on latest stable major.

**DR-5.4 — Breaking change horizon (VUR-BP)**
For dependencies within 1 major version of a new release with announced breaking changes:
- Flag: "Breaking change in vX.0 — review migration guide before upgrade"
- Especially: FastAPI v0.x → v1.x migration, SQLAlchemy 1.x → 2.x, Pydantic v1 → v2 (different APIs)

**DR-5.5 — Security patch currency (VUR-PL)**
Any available security patch not applied: MEDIUM RISK minimum, upgrades to HIGH if patch addresses CVSS ≥7.0.

**Scoring:**
- Floating `latest` in production Docker image: -3 pts each
- EOL dependency in production path: -2 pts each
- 2+ major versions behind on security-critical dep: -2 pts each
- Floor: 2/10

### DIMENSION DR-6: SUPPLY CHAIN INTEGRITY
**Laws:** Section 151.10, CD-D (artifact signing), VULN-I | **Weight:** 8%

**DR-6.1 — Artifact signing (SCR-UA)**
All artifacts must be signed and verifiable:
- Docker images: must have `sha256` digest in all `FROM` statements

- NPM packages: `package-lock.json` with integrity hashes
- Helm charts: chart digest verification in CI
Missing signing: HIGH RISK per artifact type

**DR-6.2 — Trusted registry enforcement (SCR-UB)**
All packages must be sourced from authorized registries only:
- Python: `pypi.org` only (no VCS installs without review)
- NPM: `registry.npmjs.org` only (no `git+https://` dependencies in production)

Any package sourced from unverified/unofficial registry: HIGH RISK

Scan package names for common typosquat patterns:
- Distance-1 edit from popular packages (e.g., `reqests` instead of `requests`)

- Any package with <100 downloads/week from major ecosystem: flag for review

**DR-6.4 — Abandoned package detection (SCR-AP)**
For all open-source dependencies:
- Last release > 18 months ago: MEDIUM RISK
- Last release > 36 months ago AND no stated maintenance mode: HIGH RISK
- Last release > 5 years ago AND no stated archival: HIGH RISK + recommend replacement

**DR-6.5 — Transitive dependency chain verification**

**DR-6.6 — CI/CD supply chain (CD-D)**

- GitLab CI image references must use digest
- Any `@latest` or `@main` in CI step: HIGH RISK (CI poisoning attack vector)

**Scoring:**
- Unsigned Docker image in prod: -2 pts each
- Unverified registry source: -3 pts each
- CI step on mutable tag: -1.5 pts each
- Floor: 2/10

## SECTION 13 — DEPENDENCY GOVERNANCE PRINCIPLES (SEALED)

These principles govern how the platform must treat all dependencies as governed objects, not neutral tools. Sourced from platform law.

### 13.1 The Seven Dependency Laws (from Vendor Lock-In Law)
1. **No dependency without exit**: Every integration must have a documented exit plan (R56.11)
2. **No proprietary without abstraction**: Every vendor SDK must sit behind an interface (VLR-SC)
3. **No SaaS without DPA**: Every data-handling vendor must have a signed DPA (CRR-DP)
4. **No critical path without fallback**: Every critical external dependency must have a fallback (ACR-NF)
5. **No domain inheritance**: A vendor in Domain A must not force the same vendor in Domain B (VLR-DX)
6. **No floating version in production**: Every production dependency must be pinned exactly (VUR-UP)
7. **No unscanned artifact**: Every dependency must appear in SBOM and CVE scan (VULN-I)

### 13.2 Dependency Addition Protocol
Any new dependency added to the platform must pass:
1. CVE scan — no critical/high CVEs in added version
2. License check — must be permissive or weak copyleft compatible with commercial SaaS
3. Exit plan — exit/replacement documented before merging
4. Abstraction — interface layer created if vendor-specific SDK
5. Fallback — fallback vendor or graceful degradation documented
6. SBOM — added to SBOM before any code using it is merged
Dependency merged without passing all 6 gates: HIGH RISK per missing gate

### 13.3 Dependency Removal Protocol
When removing a dependency:
1. Confirm no code still imports it
2. Remove from all manifests (not just primary)
3. Update SBOM
4. Remove from CI/CD caches
5. If the dependency handled data, verify data portability completed
Incomplete removal: MEDIUM RISK (ghost dependency risk)

### 13.4 BSL/License-Watch List
The following technologies require active monitoring for license changes:
- HashiCorp Vault → Use **OpenBao** or **Vault OSS** (pre-BSL fork)
- Terraform → Use **OpenTofu** (declared in platform stack) ✅
- Elasticsearch → Use **OpenSearch** (declared in platform stack) ✅
- MongoDB → Use PostgreSQL or check current SSPL status
- Redis → Redis 7.x is still BSD-3 for OSS version (watch for commercial changes)
- Grafana → Apache 2.0 (watch for any BSL migration signals)
